# <a name="reprotect-failed-over-azure-vms-to-the-primary-region"></a>重新保护已故障转移到主要区域的 Azure VM

使用[Azure 站点恢复](site-recovery-overview.md)将 Azure VM 从一个区域[故障转移到](site-recovery-failover.md)另一个区域时，VM 将启动到辅助区域，处于**未受保护的**状态。 如果要将 VM 故障退回主区域，执行以下任务：

1. 在次要区域中重新保护 VM，以便这些 VM 开始复制到主要区域。
1. 重新保护完成后，VM 正在复制，则可以从辅助区域故障转移到主区域。

## <a name="prerequisites"></a>先决条件

- 必须提交将 VM 从主要区域故障转移到次要区域的操作。
- 主要目标站点应该可用，并且应该能够在该区域中访问或创建资源。

## <a name="reprotect-a-vm"></a>重新保护 VM

1. 在**Vault** > **复制项目中**，右键单击 VM 上的失败，然后选择 **"重新保护**"。 重新保护方向应该显示为从次要区域到主要区域。

   ![重新保护](./media/site-recovery-how-to-reprotect-azure-to-azure/reprotect.png)

1. 查看资源组、网络、存储和可用性集。 然后单击“确定”****。 如果有任何资源标记为新资源，则这些资源将作为重新保护过程的一部分创建。
1. 重新保护作业使用最新数据植入目标站点。 作业完成后，将进行增量复制。 然后，可以故障转移回到主要站点。 可以使用自定义选项，选择要在重新保护期间使用的存储帐户或网络。

   ![自定义选项](./media/site-recovery-how-to-reprotect-azure-to-azure/customize.png)

### <a name="customize-reprotect-settings"></a>自定义重新保护设置

您可以在重新保护期间自定义目标 VM 的以下属性。

![自定义](./media/site-recovery-how-to-reprotect-azure-to-azure/customizeblade.png)

|properties |说明  |
|---------|---------|
|目标资源组 | 修改创建 VM 的目标资源组。 在重新保护的过程中，将删除目标 VM。 可以选择在故障转移后要在其下创建 VM 的新资源组。 |
|目标虚拟网络 | 执行重新保护作业期间，无法更改目标网络。 若要更改网络，请重做网络映射。 |
|目标存储（辅助 VM 不使用托管磁盘） | 可以更改故障转移后 VM 使用的存储帐户。 |
|副本托管磁盘（辅助 VM 使用托管磁盘） | Site Recovery 在主要区域中创建副本托管磁盘以监视辅助 VM 的托管磁盘。 |
|缓存存储 | 可以指定复制期间要使用的缓存存储帐户。 默认情况下，如果不存在，则会创建新的缓存存储帐户。 |
|可用性集 | 如果次要区域中的 VM 属于可用性集，则可为主要区域中的目标 VM 选择可用性集。 默认情况下，Site Recovery 会尝试在主要区域中查找并使用现有的可用性集。 在自定义期间，可以指定新的可用性集。 |

### <a name="what-happens-during-reprotection"></a>重新保护期间会发生什么情况？

默认情况下，会发生以下情况：

1. 在运行故障转移的 VM 的区域中创建一个缓存存储帐户。
1. 如果目标存储帐户（主要区域中的原始存储帐户）不存在，则会创建一个新帐户。 分配的存储帐户名称是辅助 VM 使用的存储帐户的名称，后缀为`asr`。
1. 如果 VM 使用托管磁盘，则主要区域中会创建副本托管磁盘以存储从辅助 VM 磁盘中复制的数据。
1. 如果目标可用性集不存在，则在必要时作为重新保护作业的一部分创建一个新可用性集。 如果已自定义重新保护设置，则使用所选集。

触发重新保护作业时，如果目标 VM 存在，则会发生以下情况：

1. 关闭正在运行的目标端 VM。
1. 如果 VM 使用托管磁盘，则使用`-ASRReplica`后缀创建原始磁盘的副本。 原始磁盘将被删除。 副本`-ASRReplica`用于复制。
1. 如果 VM 使用的是非托管磁盘，则会分离目标 VM 的数据磁盘并将其用于复制。 将会创建 OS 磁盘的一个副本并将其附加到 VM 上。 将分离原始 OS 磁盘并将其用于复制。
1. 仅同步源磁盘与目标磁盘之间的更改。 将通过比较两个磁盘来计算差异，然后传输这些差异。 检查下面，查找完成重新保护的估计时间。
1. 同步完成后，增量复制将开始，并且根据复制策略创建恢复点。

当您触发重新保护作业，并且目标 VM 和磁盘不存在时，将发生以下情况：

1. 如果 VM 使用托管磁盘，则使用`-ASRReplica`后缀创建副本磁盘。 副本`-ASRReplica`用于复制。
1. 如果 VM 使用的是非托管磁盘，则会在目标存储帐户中创建副本磁盘。
1. 将全部磁盘从故障转移的区域复制到新的目标区域。
1. 同步完成后，增量复制将开始，并且根据复制策略创建恢复点。

#### <a name="estimated-time-to-do-the-reprotection"></a>估计的进行重新保护所需的时间

在大多数情况下，Azure 站点恢复不会将完整数据复制到源区域。
以下条件确定复制的数据量：

1. 如果源 VM 数据由于某种原因（如资源组更改/删除）被删除、损坏或无法访问，则在重新保护期间将发生完整的初始复制，因为源区域上没有可用的数据可供使用。
1. 如果源 VM 数据是可访问的，则仅通过比较磁盘然后传输来计算差异。 检查下表以获取估计时间。

|示例情况 | 重新保护所需要时间 |
|---|---|
|源区域具有 1 个 VM，具有 1 个 TB 标准磁盘。<br/>仅使用 127 GB 数据，其余磁盘为空。<br/>磁盘类型是标准磁盘类型，吞吐量为 60 MiB/S。<br/>故障转移后无数据更改。| 大致时间：45 分钟 = 1.5 小时。<br/>在重新保护期间，站点恢复将填充所有数据的检查和，这些数据需要 127 GB/45 MB，大约 45 分钟。<br/>站点恢复自动缩放需要一些开销时间，大约 20-30 分钟。<br/>无出口费用。 |
|源区域具有 1 个 VM，具有 1 个 TB 标准磁盘。<br/>仅使用 127 GB 数据，其余磁盘为空。<br/>磁盘类型是标准磁盘类型，吞吐量为 60 MiB/S。<br/>故障转移后 45 GB 数据更改。| 大致时间：1 小时 = 2 小时。<br/>在重新保护期间，站点恢复将填充所有数据的检查和，这些数据需要 127 GB/45 MB，大约 45 分钟。<br/>传输时间以应用 45 GB 的更改，即 45 GB/45 MBps，大约 17 分钟。<br/>出口费用将用于 45 GB 数据更改，而不是校验和。 |

当 VM 在故障返回主区域（即，如果 VM 从主区域重新保护到 DR 区域）后重新受到保护时，将删除目标 VM 和关联的 NIC（s）。

当 VM 从 DR 区域重新保护到主区域时，我们不会删除前主 VM 和关联的 NIC。

## <a name="next-steps"></a>后续步骤

保护 VM 后，可以启动故障转移。 故障转移将关闭辅助区域中的 VM，并在主区域中创建和引导 VM，在此过程中会短暂停机。 我们建议您为此过程选择适当的时间，并在启动对主站点的完全故障转移之前运行测试故障转移。 [了解有关](site-recovery-failover.md)Azure 站点恢复故障转移的更多详细信息。

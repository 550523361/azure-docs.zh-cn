# <a name="best-practices-for-creating-assessments"></a>创建评估最佳实践

[Azure Migrate](migrate-overview.md)提供一个中心工具，可帮助你发现、 评估和将应用程序、 基础结构和工作负荷迁移到 Microsoft Azure。 在中心包括 Azure 迁移工具和第三方独立软件供应商 (ISV) 产品/服务。 

创建使用 Azure Migrate Server 评估工具评估时，本文汇总了最佳实践。 

## <a name="about-assessments"></a>有关评估

使用 Azure Migrate Server 评估创建的评估是数据的时间点快照。 有两种类型的 Azure Migrate 中评估。

**评估类型** | **详细信息** | **数据**
--- | --- | ---
**基于性能的** | 提供基于收集的性能数据的建议的评估 | VM 大小建议基于 CPU 和内存利用率数据。<br/><br/> 磁盘类型建议 （标准或高级托管磁盘） 基于的 IOPS 和吞吐量的本地磁盘。
**作为-本地** | 不使用的性能数据提供建议的评估。 | VM 大小建议基于在本地 VM 大小<br/><br> 建议的磁盘类型取决于您选择了用于评估设置的存储类型。

### <a name="example"></a>示例
例如，如果您有 20%的利用率，并使用 10%的利用率，8 gb 内存的 4 个内核的本地 VM 评估将，如下所示：

- **基于性能的评估**:
    - 建议内核和内存基于 core （0.8 核心） 和 (0.8 GB) 内存使用率。
    - 评估将应用默认的舒适因子的 30%。
    - VM 建议： ~1.4 内核 (0.8 x1.3) 和 ~1.4 GB 内存。
- **作为-是 （本地） 评估**:
    -  建议使用四个核心; 的 VM8 GB 的内存。

## <a name="best-practices-for-creating-assessments"></a>创建评估最佳实践

Azure Migrate 设备持续配置你的本地环境，并将元数据和性能数据发送到 Azure。 请遵循这些最佳实践创建评估：

- **为创建的是评估**:您可以创建作为-立即发现之后是评估。
- **创建基于性能的评估**:设置后发现，我们建议你至少等待一天，然后再运行基于性能的评估：
    - 收集性能数据需要时间。 至少等待一天可以确保在运行评估之前有足够的性能数据点。
    - 性能数据的设备收集每个性能指标，每隔 20 秒的实时数据点，并回滚事务最多五分钟的单个数据点。 设备会发送到 Azure 评估计算每隔一小时的五分钟数据点。  
- **获取最新数据**:评估不会自动更新最新的数据。 若要使用的最新数据更新评估，需要重新运行它。 
- **请确保持续时间匹配**:当您运行的基于性能的评估时，请确保您的个人资料评估持续时间的环境。 例如，如果性能持续时间设置为一周创建评估，你需要等待至少一周后启动发现，所有数据点收集。 如果不这样做，评估不会获得的五星分级。 
- **避免缺失的数据点**:在缺少基于性能的评估中的数据点可能会导致以下问题：
    - 在评估期间关闭 Vm 并不会收集性能数据。 
    - 如果在基于性能历史记录该月期间创建 Vm。 这些 Vm 的数据将不超过一个月。 
    - 发现之后立即, 创建评估或评估时间与性能数据收集时间不匹配。

## <a name="best-practices-for-confidence-ratings"></a>置信度评级的最佳做法

当运行基于性能的评估时，从 1 星型 （最低） 到 5 星 （最高） 分级的置信度发放给评估。 若要使用置信度评级有效：
- Azure Migrate Server 评估的 VM CPU/内存和磁盘 IOPS/吞吐量数据需要的利用率数据。
- 每个网络适配器附加到 VM，Azure Migrate 需要网络输入/输出数据。
- 如果使用率数据不可用在 vCenter Server 中，Azure migrate 提供的大小建议可能不是可靠的。 

具体取决于可用的数据点的百分比下, 表总结了评估的置信度分级。

   **数据点可用性** | **置信度分级**
   --- | ---
   0%-20% | 1 星
   21%-40% | 2 星
   41%-60% | 3 星
   61%-80% | 4 星
   81%-100% | 5 星

- 如果您收到低于 5 星的评估的置信度分级，至少等待一天，然后重新计算评估。
- 低级别意味着，则可能无法可靠重设大小建议。 在这种情况下，我们建议您修改评估属性，以用作-是在本地评估。

## <a name="common-assessment-issues"></a>评估常见问题

下面介绍了如何解决一些常见环境的问题，会影响评估。

###  <a name="out-of-sync-assessments"></a>带同步评估

如果添加或从组中删除计算机创建评估后，将标记为你创建的评估**扩展同步**。运行评估再次 (**重新计算**) 以反映组更改。

### <a name="outdated-assessments"></a>过时的评估

如果有已评估的组中的 Vm 的本地更改，标记评估**过时**。 以反映所做的更改，运行重新评估。

### <a name="missing-data-points"></a>缺失的数据点

评估可能不具有多种原因导致的所有数据点：

- 在评估期间可能会关闭 Vm 并不会收集性能数据。 
- 可能在其上的性能历史记录所基于的月份期间创建 Vm，因此其性能数据是不超过一个月。 
- 发现之后立即创建评估。 若要指定的时间内收集的性能数据，你需要等待指定在运行评估之前的时间量。 例如，如果你想要评估一周的性能数据，需要等待一周后发现。 如果不这样做评估不会获得的五星分级。

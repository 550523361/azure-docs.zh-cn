# <a name="azure-files-share-snapshot-preview-overview"></a>Azure 文件共享快照（预览版）概述
Azure 文件提供了获取文件共享的共享快照的功能。 共享快照（预览版）可以捕获在某个时间点的共享状态。 在本文中，我们将介绍共享快照提供的功能，以及如何在自定义用例中加以利用。


## <a name="when-to-use-share-snapshots"></a>何时使用共享快照

### <a name="protection-against-application-error-and-data-corruption"></a>防止应用程序错误和数据损坏
-------------------------------------------------------------------------

使用 Azure 文件共享的应用程序会执行各种操作，如编写、读取、存储、传输或处理。 因此，应用程序可能会被错误配置，或者误引入 bug，进而导致意外覆盖或损坏几个块。 为防止出现这些情况，你可以在部署新应用程序代码之前截取共享快照，如果在新部署中引入了 bug 或应用程序错误，则可以返回到驻留在该文件共享中之前版本的数据。 

### <a name="protection-against-accidental-deletions-or-unintended-changes"></a>防止意外删除或意外更改
----------------------------------------------------------------------------------------------

假设用户正在处理驻留在文件共享中的文本文件。 一旦文本文件关闭，将无法撤消我们做的更改。 在这些情况下，将需要恢复文件的前一个版本。 通过共享快照，即可在文件被意外重命名或删除时将其恢复至先前的版本。

### <a name="general-backup-purposes"></a>常规备份目的
---------------------------

创建 Azure 文件共享后，可以定期创建 Azure 文件共享的共享快照，以便使用它进行数据备份。 定期拍摄共享快照，有助于维护之前版本的数据，以备未来审计之需或灾难恢复之用。

## <a name="share-snapshot-capabilities"></a>共享快照功能

共享快照是数据在一个时间点只读副本。 可以使用 REST API 创建、删除和管理快照。 此外，客户端库、CLI 和 Azure 门户中也提供了相同的功能。 共享快照的 PowerShell 集成也即将推出。 你可以使用 REST API 和 SMB 查看共享快照。 客户可以检索目录或文件的版本列表，也可以直接作为驱动程序装载特定版本。 在创建共享快照后，可以读取、复制或删除该快照，但无法对其进行修改。 无法将整个共享快照复制到另一个存储帐户，你必须使用 azcopy 或其他复制机制逐个复制文件。

共享快照功能是在文件共享级别上提供的，而检索是在单个文件级别提供的，可用于恢复单个文件。 你可以使用 SMB、REST API、门户、客户端库，或者使用 PowerShell/CLI 工具还原一个完整的文件共享。

文件共享的共享快照与其基本文件共享相同，不过，共享 URI 的后面追加了一个 DateTime 值，用于指示共享快照的生成时间。 例如，如果一个文件共享 URI 是 http://storagesample.core.file.windows.net/myshare，则共享快照 URI 将类似于
```
http://storagesample.core.file.windows.net/myshare?snapshot=2011-03-09T01:42:34.9360000Z.
```

除非显式删除，否则共享快照会一直保留。 共享快照的生存期不能长于其基本文件共享。 可以枚举与基本文件共享相关联的快照，以跟踪当前快照。 创建文件共享的共享快照时，驻留在共享系统属性中的文件会被复制到具有相同值的共享快照中。 基本文件和文件共享的元数据也会复制到共享快照，除非在创建共享快照时为其指定了不同的元数据。

请先删除所有共享快照，否则无法删除具有共享快照的共享。


## <a name="share-snapshot-space-usage"></a>共享快照空间使用情况 

共享快照本质上是增量的，这意味着只会保存最新共享快照之后发生更改的数据。 这将减少创建共享快照所需的时间，并节省存储成本。 对象或属性或元数据更新操作的任何写入操作，都将被计入“内容更改”，并存储在共享快照中。 

为了节省空间，可以在改动幅度最大的期间删除共享快照。

尽管共享快照是增量方式进行保存的，但共享快照删除过程也是有意如此设计，以便只需保留最新的共享快照即可还原共享。 删除共享快照时，仅删除该共享快照所独有的数据。 活动快照包含浏览和还原数据（从拍摄共享快照开始）到原始位置或替代位置所需的所有信息。 可以在项级别进行还原。

快照不计入 5 TB 共享限制。 共享快照占用的总空间没有限制。 存储帐户限制仍然适用。

## <a name="limits"></a>限制

Azure 文件目前允许的共享快照的上限是 200 个。 在 200 个共享快照之后，用户必须删除旧的共享快照，以便创建新的共享快照。 对创建共享快照的同时调用没有限制。
特定文件共享所能占用的共享快照空间没有限制。 

## <a name="copy-data-back-to-a-share-from-share-snapshot"></a>将数据从共享快照复制回共享

涉及文件和共享快照的复制操作遵循以下规则：

可以将文件共享快照中的各个文件复制到其基本共享或任何其他位置。 通过从共享快照中逐个复制文件，可以还原文件的早期版本或还原完整的文件共享。 共享快照不会被提升到基本共享。 复制后共享快照保持不变，但基本文件共享将被共享快照中可用的数据副本覆盖。 所有还原后的文件会计入“已更改内容”。

可将共享快照中的文件复制到具有不同名称的目标。 生成的目标文件是可写文件，而不是共享快照。

使用副本覆盖目标文件时，与原始目标文件关联的所有共享快照均将保持不变。

## <a name="general-best-practices"></a>常规最佳做法 

在 Azure 上运行基础结构时，尽可能自动备份数据恢复。 自动化操作比手动流程更可靠，有助于提高数据保护和可恢复性。 可以使用 REST API、客户端 SDK 或脚本来实现自动化。

在部署共享快照计划程序之前，仔细考虑共享快照频率和保留设置，以免产生不必要的共享快照收费。

共享快照只提供文件级保护，文件共享或存储帐户中的任何意外删除不受共享快照的保护。 可以锁定存储帐户或资源组，以保护存储帐户不被意外删除。

# <a name="azure-marketplace-listing-guidelines"></a>Azure 市场列表准则

本文档包含 Azure Marketplace 中列出的有关新产品/服务和服务的要求和指导原则。

所有产品/服务必须满足 "列出所有产品的要求" 部分中列出的要求。

针对特定列表类型提供了其他要求和清单，其中包括： 
* [试用套餐要求](#trial-offer-requirements)
* [SaaS 应用要求](#saas-app-requirements)
* [容器套餐要求](#container-offer-requirements)
* [咨询套餐要求](#consulting-offer-requirements)

## <a name="listing-requirements-for-all-offers"></a>列出所有产品/服务的要求

|  | 列表元素 | 基本要求 | 最佳要求 |
|:--- |:--- |:--- |:--- |
| 1 | 潜在客户目标 | 已配置潜在客户目标。| 一个商业合作伙伴（OCP）目录具有 "合作伙伴解决方案" 选项卡中也列出的潜在客户目标 CRM 信息。 | 
| 2 | 套餐标题 | 简要介绍解决方案产品/服务。 与解决方案在合作伙伴网站上的在线升级相匹配。 | 包含关键字搜索词。 |
| 3 | 徽标 | 徽标正确显示。 | 徽标正确显示，并包含一个 "英雄图像"，该图像是 Azure 门户中的大型格式的图像。 |
| 4 |  套餐说明 | <ul><li> 包含2-3 段。</li><li>可以一目了然地理解解决方案产品/服务。</li><li>无拼写和语法错误。</li><li>是全面的，可捕获目标受众、用户类型以及有价值的原因（价值主张）。</li><li>在段落叙述形式中，具有易于理解的短句子。</li></ul> | <ul><li> 概述了目标行业（如果相关）。</li><li>好的样式格式，每个段落标题都包含一个句子或短语，用于汇总后面的内容并使用项目符号（如果需要）来强调主要优点。 其目的是让读者一目了然地了解该产品/服务，无需阅读长段。</li><li>每个段落之间有间距。 它看起来像汽车手册。 也就是说，它是全面的，只是不带技术术语的产品。 | 
| 5 | 类别和行业 | <ul><li>类别与解决方案产品的功能相匹配。</li><li>不要扩展到解决方案不适合的类别。</li><li>最佳行业或所选的所有行业（如果并非最佳行业）。 | <ul><li>最大 3个类别。</li><li>最大 3个行业。 |
| 6 | 图像 | <ul><li>不需要图像，但如果提供，则必须将其显示为高分辨率图像。</li><li>映像要求在云合作伙伴门户（CPP）和此处列出。</li><li>屏幕截图中包含的文本会清晰明了，图像清晰。 | 可以轻松理解解决方案产品/服务。 |
| 7 | 视频 | <ul><li>无需视频，但如果提供，则必须播放，且不会出现任何错误。</li><li>如果提供，则它可能不会引用竞争者公司，*除非*它演示迁移解决方案。 |<ul><li>理想情况下，长度为3分钟或更长。</li><li>可以通过视频内容轻松理解解决方案产品/服务。</li><li>提供解决方案功能的演示。 |
| 8 | 列出状态（调用操作） | <ul><li>必须标记为以下类型之一： <ul><li>*与我联系*</li><li>*试用*/*获取试用版现在*/*开始试用*/*测试驱动器*</li><li>*立即购买*立即*获取*/</li></ul></ul> | 客户可随时了解后续步骤： <ol><li>试用试用版。</li><li>立即购买。</li><li>通过电子邮件或电话号码联系，以安排概念证明（POC）、评估或简报。</li></ol> | 
| 9 | 解决方案定价 | 必须具有解决方案定价选项卡/详细信息，定价必须是合作伙伴解决方案产品/服务的本地货币。 | 要提供客户选项，应提供多个计费选项。 |    
| 10 | 了解详细信息 | 底部的链接（在描述下，而不是左侧的 Azure Marketplace 链接）将导致解决方案的详细信息，并可公开提供并正确显示。 |    指向特定项（例如，合作伙伴网站上的规范页）而不只是合作伙伴主页的链接。 | 
| 11 | 解决方案支持和帮助 | 链接到下面至少一项： <ul><li>电话号码</li><li>电子邮件支持</li><li>聊天代理</li><li>社区论坛 |<ul><li>列出了所有支持方法。</li><li>在*试用*或*测试驱动器*期间免费提供付费支持。 | 
| 12 | 合法 | 策略或条款可通过公共 URL 获得。 |  |

## <a name="saas-app-requirements"></a>SaaS 应用要求

|  | 列表元素 | 基本要求 | 最佳要求 | 
|:--- |:--- |:--- |:--- |
| 1 | 套餐标题 |<ul><li>必须仅包含小写字母、字母数字字符、短划线或下划线。 标题在发布后无法修改。</li><li>描述解决方案产品。</li><li>与合作伙伴网站上的解决方案在线升级。 | 包含关键字搜索词。 |
| 2 | 技术信息：配置 |<ul><li>对于软件即服务（SaaS）应用，请选择是仅列出你的应用，还是让客户通过 Azure 购买你的应用。</li><li>在产品/服务的 "购置" 按钮上选择所需的文本：*免费**版、免费试用版*或 "*与我联系*"。</li><li>在弹出窗口中，如果应用使用技术，请仅选择一个适用的产品： Cortana Intelligence、Power BI 解决方案模板或电源应用。 |  |
| 3 | 体验版 | 选择 *"是" 或 "* *否*" | 客户可随时了解后续步骤： <ol><li>试用试用版。</li><li>立即购买。</li><li>通过电子邮件或电话号码联系以安排 POC、评估或简报。</li></ol> |
| 4 | 店面详细信息：产品/服务摘要 | 显示在应用的搜索页面上，最多包含100个字符。 | |
| 5 | 店面详细信息：行业 | 行业（最多2个）：选择你的应用程序最佳对齐并适用的行业。 | |
| 6 | 套餐说明 | <ul><li>允许使用简单的 HTML， \<其中包括 p \<>、em \<>、ul \<>、li \<>、ol> 和标头标记。 最大 3000个字符。</li><li>包含2-3 段。</li><li>可以一目了然地理解解决方案产品/服务。</li><li>说明是全面的，可捕获目标受众、用户的类型以及它的价值（价值主张）。</li><li>采用段落叙述形式，具有易于理解的短句子。 | <ul><li>概述目标行业（如果相关）。</li><li>好的样式格式，每个段落标题都包含一个句子或短语，用于汇总后面的内容并使用项目符号（如果需要）来强调主要优点。 其目的是让读者一目了然地了解该产品/服务，无需阅读段落。</li><li>每个段落之间有间距。 它看起来像汽车手册。 也就是说，它是全面的，只是不带技术术语的产品。 | 
| 7 | 营销项目 | 徽标正确显示。 | <ul><li>徽标包含一个 "多内容图像"，该图像是 Azure 门户中的大型格式的图像。</li><li>徽标：小（48 &#215; 48 像素）和大（216 &#215; 216 像素）。</li><li>[英雄图像要求](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-new-azure-apps-offer)。</li><li>屏幕截图（最多5个）： CPP 需要分辨率为 1280 &#215; 720 像素的 PNG 图像。 |
| 8 | 类别和行业 | <ul><li>类别与解决方案产品功能匹配。</li><li>在弹出窗口中选择至少一项。</li><li>不要扩展到解决方案不适合的类别。</li><li>最佳行业或所选的所有行业（如果并非最佳行业）。 | 最大 选择3个类别（如果适用）。 |
| 9 | 潜在顾客管理 | 选择用于存储潜在顾客的系统。 请参阅[获取客户主管](./partner-center-portal/commercial-marketplace-get-customer-leads.md)以连接 CRM 系统。 |  |
| 10 | 联系人：解决方案支持和帮助 | <ul><li>工程联系人姓名：应用的工程联系人姓名。 此联系人将收到来自 Microsoft 的技术邮件。</li><li>工程联系人电子邮件：应用的工程联系人电子邮件地址。</li><li>工程联系人电话：工程联系人的电话号码。 支持[ISO 电话号码表示法](https://en.wikipedia.org/wiki/E.123)。</li><li>支持联系人姓名：应用的支持联系人的姓名。 此联系人将收到来自 Microsoft 的支持相关邮件。</li><li>支持联系人电子邮件：应用的支持联系人的电子邮件地址。</li><li>支持联系人电话：支持联系人的电话号码。 支持[ISO 电话号码表示法](https://en.wikipedia.org/wiki/E.123)。</li><li>支持 URL：支持页的 URL。 | <ul><li>列出了所有支持方法。</li><li>在*试用*或*测试驱动器*期间免费提供付费支持。 |
| 11 | 合法 |<ul><li>隐私策略 URL： CPP 的 "隐私策略 URL" 字段中的应用隐私策略的 URL。</li><li>使用条款：应用的使用条款。 客户必须接受这些条款才能试用应用。 |    策略或条款可通过公共 URL 网站获得。 |

## <a name="consulting-offer-requirements"></a>咨询套餐要求

|  | 列表元素 | 基本要求 | 最佳要求 |
|:--- |:--- |:--- |:--- |
| 1 | 套餐标题  |  必须以格式*名：持续时间类型*清楚列出服务类型和持续时间（例如，"产品/服务：1周的概念证明"）。  | <ul><li>不重复发布服务器名称。</li><li>标题反映了说明中已经提供的标题样式。 避免使用长标题。 | 
| 2 | 套餐说明  |<ul><li>确保正确使用[Microsoft 产品名称](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx)。</li><li>标记为价格的产品 *：估计*值必须在顶部或底部具有注释，以说明可变性（客户端、正在迁移的服务器的数量等）。</li><li>每种产品/服务类型都具有说明要求，如下所示： <ul><li>简报至少需要四个或五个项目符号，其中包含有关要涵盖的主题的信息。</li><li>所有研讨会都需要一则安排。</li><li>所有议程都必须按天或按周细分，具体取决于研讨会的持续时间。</li><li>评估、POC、非培训讨论会和实施服务需要可交付结果。</li><li>培训讨论会不需要可交付结果，但他们需要更详细的议程，其中包含将涵盖的主题。</li></ul></ul> | <ul><li>任何产品/服务都具有议程和交付成果。</li><li>产品/服务包括一个段落，其中包含有关在上一节中提供该服务的公司的上下文。</li><li>产品/服务中包含有关服务本身的值的段落。 |
| 3 | Markdown 格式设置  | 所有产品/服务都必须使用 Markdown 格式，以便在将产品/服务转换为 HTML 时正确呈现。  |  |
| 4 | 类别和行业 | 类别不相关。 |  | 
| 5 | 列出状态（调用操作） | 自动列出为 "*与我联系*"。 |  | 
| 6 | 解决方案支持和帮助  | 不需要支持和帮助。  | | 
| 7 | 隐私策略和可用使用条款  |  不需要策略或条款。  | | 
| 8 | 服务类型 | 确保服务类型与标题匹配。 | |
| 9 | 能力 | 至少必须有以下其中一项的资格：<ul><li>应用程序开发</li><li>应用程序集成</li><li>Application Lifecycle Management</li><li>云平台<li>数据分析</li><li>数据中心</li><li>数据平台</li><li>DevOps |
| 10 | Products | 必须是 Azure 产品。 | |
| 11 | 国家/地区 | 确保国家/地区与所选货币匹配。 | |
| 12 | 了解详细信息  | <ul><li>底部的链接（在描述下，而不是左侧的 Azure Marketplace 链接）将导致解决方案的详细信息，并且这些信息公开提供并正确显示。</li><li>链接必须具有 "友好名称"，并且不会显示为任何下载的文件名。 | |

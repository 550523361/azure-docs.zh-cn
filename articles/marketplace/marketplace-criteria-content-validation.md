# <a name="azure-marketplace-listing-guidelines"></a>Azure 市场列表准则

本文档包含要求准则，以及审查者在审查列入 Azure 市场中的新套餐和服务时所遵循的查检表。

所有套餐必须符合下面所述的[所有列表要求](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#all-listing-requirements)。 对于特定的列表类型，提供了额外的要求和查检表，包括：附有[试用版](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#trial-offer-requirements)、[SaaS 应用](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#saas-app-requirements)、[容器](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#container-offer-requirements)和[咨询服务](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#consulting-offer-requirements)的套餐。


## <a name="all-listing-requirements"></a>所有列表要求

|  | 列表元素 | 基本要求 | 最佳要求 |
|:--- |:--- |:--- |:--- |
| 1 | 潜在客户目标 | 已配置潜在顾客目标| 合作伙伴解决方案选项卡中还列出了 OCP 目录的潜在顾客目标 CRM 信息 | 
| 2 | 套餐标题 | 描述解决方案套餐。 在合作伙伴的网站上匹配解决方案的在线推广 | 包含搜索关键字 |
| 3 | 徽标 | 徽标正常显示 | 徽标显示正确，并在 Azure 门户中包含 [英雄图像，这是大格式图像 |
| 4 |  套餐说明 | <br>• 2-3 段 <br>• 解决方案产品一目了然，很容易理解 <br>* 提供描述是没有拼写和语法错误 <br>• 优惠描述是全面的和捕获的：目标受众，用户类型，为什么它的价值（价值道具） <br>* 报价描述是段落叙述形式与短句，易于理解 | <br>• 目标行业概述（如果相关） <br>• 良好的样式格式，每个段落标题都有一个句子或短语，汇总以下内容，并包括要点（适当时），以强调关键优势。 此处的目标是使读者无需阅读整个段落，就能通过格式快速轻松地了解套餐。 <br>• 每个段落之间都有间距，读起来就像汽车宣传册一样。 这意味着不需要插入技术特性或代码术语，而只是以简单的字词全面描述套餐。 | 
| 5 | 类别和行业 | <br>• 类别匹配解决方案提供功能 <br>• 不要扩展到解决方案不适合的类别 <br>• 最佳行业或所选所有行业（如果不是最佳行业） | <br>• 最大。 3 个类别 <br>• 最大。 3 个行业 |
| 6 | 映像 | <br>• 无需图像，但如果提供，则必须显示高分辨率图像。 <br>• 图像要求列在合作伙伴中心，也在这里 <br>• 屏幕截图中包含的文本清晰可辨，图像清晰 | 能够快速轻松地了解解决方案套餐 |
| 7 | 视频 | <br>• 无需视频，但如果提供，则必须播放没有任何错误。 <br>• 如果提供，不得指竞争对手公司，除非演示迁移解决方案 |<br>• 理想情况下 3 分钟或更长时间 <br>• 解决方案提供通过视频内容轻松理解的解决方案 <br>• 解决方案功能演示 |
| 8 | 列表状态（行动号召） | <br>必须标记以下类型之一： <ul>o 与我联系</ul><ul>o 试用/立即试用/开始试用/体验版</ul><ul>o 立即购买/立即购买</ul> | 客户马上能够了解后续步骤是什么： <ul>1. 试用试用版</ul> <ul>2. 立即购买</ul><ul>3. 通过电子邮件或电话号码联系，安排 PoC、评估或简报。</ul> | 
| 9 | 解决方案定价 | 必须提供解决方案定价选项卡/详细信息，并采用合作伙伴解决方案套餐的本地货币 | 提供多个分层定价计费选项供客户选择 |    
| 10 | 了解详细信息 | 底部的链接（在描述下，而不是左侧的市场链接）导致有关解决方案的详细信息，并且公开可用并正确显示 |    指向特定项目的链接（例如，合作伙伴网站上的规格页面），而不仅仅是合作伙伴主页网站 | 
| 11 | 解决方案支持和帮助 | 链接到下面至少一项： <br>o 电话号码 <br>o 电子邮件支持 <br>o 聊天代理 <br>o 社区论坛 |<br>• 列出的所有支持方法：电话&电子邮件支持。 <br>• 在试驾期间免费提供付费支持 | 
| 12 | 合法 | 通过公共 URL 提供政策或条款 |  |

## <a name="saas-app-requirements"></a>SaaS 应用要求

|  | 列表元素 | 基本要求 | 最佳要求 | 
|:--- |:--- |:--- |:--- |
| 1 | 套餐标题 |<br>* 只能由小写、字母数字字符、破折号或下划线组成。 一经发布不可修改。 <br>• 解决方案提供的描述性。<br>• 在合作伙伴的网站上匹配解决方案的在线推广。 | 包含搜索关键字 |
| 2 | 技术信息：配置 |<br>• 对于 SaaS 应用，选择是只想列出应用，还是希望使客户能够通过 Azure 购买你的应用。<br>• 在优惠的获取按钮上选择您想要的相应文本：免费、免费试用或与我联系<br>• 仅选择（弹出式框）- 如果应用利用该技术，这些适用产品之一：Cortana 智能、电源 BI 解决方案模板、电源应用 |  |
| 3 | 体验版 | 选择：“是”或“否” | 客户马上能够了解后续步骤是什么： <br>1. 试用试用版 <br>2. 立即购买 <br>3. 通过电子邮件或电话号码联系，安排 PoC、评估或简报。 |
| 4 | 店面详细信息：套餐详细信息 | 此信息将显示在应用的搜索页上，最多包含 100 个字符 | |
| 5 | 店面详细信息：行业 | 行业（最多 2 个）：选择应用最适合的行业。 | |
| 6 | 产品/服务描述 | <br>• 允许使用简单的 HTML，包括 p、em、ul、li、ol 和头标记。 最多包含 3000 个字符。 <br>• 2-3 段 <br>• 解决方案产品一目了然，很容易理解 <br>* 描述是全面的和捕获：目标受众 *，用户类型，为什么它的价值（价值道具）<br>* 提供描述是段落叙述形式与短句，很容易理解。 | <br>• 目标行业概述（如果相关） <br>• 良好的样式格式，每个段落标题都有一个句子或短语，汇总以下内容，并包括要点（适当时），以强调关键优势。 此处的目标是使读者无需阅读整个段落，就能通过格式快速轻松地了解套餐。 <br>• 每个段落之间都有间距，读起来就像汽车宣传册一样。 这意味着不需要插入技术特性或代码术语，而只是以简单的字词全面描述套餐。 | 
| 7 | 营销项目 | 徽标正常显示 | <br>• 徽标在 Azure 门户中包括"英雄映像"。 必须为 815 x 290 像素。 <br>• 徽标：要求使用小型 （48x48） &大 （216x216） <br> • 屏幕截图（最大 5）：需要 .png 图像（1280 x 720 像素）。 |
| 8 | 类别和行业 | <br>• 类别匹配解决方案提供功能 <br>• 应至少从弹出式框中选择 1 个项目 <br>• 不要扩展到解决方案不适合的类别 <br>• 最佳行业或所选所有行业（如果不是最佳行业） <br>• 不要扩展到解决方案不适合的类别 <br>• 最佳行业或所选所有行业（如果不是最佳行业） | 最大 选择 3 个类别（如果适用）。 |
| 9 | 线索管理 | 选择用于存储潜在顾客的系统。 请参阅[获取客户线索](./partner-center-portal/commercial-marketplace-get-customer-leads.md)以连接 CRM 系统。 |  |
| 10 | 联系人：解决方案支持和帮助 | <br>• 工程联系人姓名：输入应用的工程联系人的名称。 此联系人将收到来自 Microsoft 的技术邮件。<br>• 工程联系人电子邮件：输入应用的工程联系人的电子邮件地址。<br>• 工程联系人 电话：输入工程联系人的电话号码。 支持 ISO 电话号码表示法；有关详细信息，请参阅 https://en.wikipedia.org/wiki/E.123。<br>• 支持联系人姓名：输入应用的支持联系人的名称。 此联系人将收到来自 Microsoft 的支持相关邮件。<br>• 支持联系人电子邮件：输入应用支持联系人的电子邮件地址。 <br>• 支持联系电话：输入支持联系人的电话号码。 支持 ISO 电话号码表示法；有关详细信息，请参阅 https://en.wikipedia.org/wiki/E.123。 <br>• 支持 URL：输入支持页面的 URL。 | <br>• 列出的所有支持方法：电话&电子邮件支持。 <br>• 在试驾期间免费提供付费支持 |
| 11 | 合法 |<br>• 隐私政策 URL：在合作伙伴中心的隐私政策 URL 字段中输入应用隐私政策的 URL。 <br>• 使用条款：输入应用的使用条款。 客户必须接受这些条款才能试用应用。 |    通过公共 URL 站点提供政策或条款 |

## <a name="consulting-offer-requirements"></a>咨询套餐要求

|  | 列表元素 | 基本要求 | 最佳要求 |
|:--- |:--- |:--- |:--- |
| 1 | 套餐标题  |  <br>• 必须以以下格式明确列出服务类型和持续时间：名称：持续时间类型。 （即"优惠参与：为期 1 周的概念证明"）  | <br>• 不重复发布者名称 <br>• 镜像说明中已有的标题样式 - 不希望长标题。 | 
| 2 | 产品/服务描述  |<br>• 确保正确使用[微软产品名称](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx) <br>• 标记为价格的报价：估计必须在优惠的顶部或底部有一个注释来解释可变性（前往客户端、迁移的服务器数量等） <br>• 每种报价类型都有如下描述要求： <ul><br>o 简报至少需要包含 4 到 5 条要点，其中提供有关简报中所述主题的信息 </ul> <ul><br>o 所有研讨会需要议程 </ul><ul><br>o 所有议程必须根据研讨会的持续时间按日期或周划分 </ul><ul><br>o 评估、概念证明、非培训研讨会、实施套餐需要交付件 </ul><ul><br>o 培训研讨会不需要可交付成果，但它们需要更详细的议程，其中将涵盖主题。 | <br>• 任何报价都有议程和可交付成果 <br>* 优惠包括一段，其中包含有关在顶部部分提供服务的公司上下文 <br>• 优惠包括一段关于服务本身作为顶部部分的价值 |
| 3 | Markdown 格式  | 所有套餐必须使用 Markdown 格式，以便在转换为 HTML 后套餐能够正常显示  |  |
| 4 | 类别和行业 | 不相关的类别 |  | 
| 5 | 列表状态 (CTA) | 自动列为“与我联系” |  | 
| 6 | 解决方案支持和帮助  | 不需要“支持和帮助”  | | 
| 7 | 提供隐私政策和使用条款  |  不需要政策或条款  | | 
| 8 | 服务类型 | 确保服务类型与标题匹配 | |
| 9 | 资质 |<br>• 必须至少为以下一项：<br>• 应用程序开发 <br>• 应用程序集成 <br>• 应用程序生命周期管理 <br>• 云平台 <br>• 数据分析 <br>• 数据中心 <br>• 数据平台 <br>• DevOps |
| 10 | Products | 必须是 Azure | |
| 11 | 国家/地区 | 确保国家和地区与所选的货币相匹配 | |
| 12 | 了解详细信息  | <br>• 底部的链接（在描述下，而不是左侧的市场链接）导致有关解决方案的详细信息，并公开提供并正确显示。 <br>• 链接必须具有"友好名称"，不得显示为任何下载的文件名 | |

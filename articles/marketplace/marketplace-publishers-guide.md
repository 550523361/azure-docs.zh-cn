# <a name="azure-marketplace-and-appsource-publisher-guide"></a>Azure Marketplace 和 AppSource 发布者指南

欢迎参阅 [Azure Marketplace](https://azuremarketplace.microsoft.com) 和 [AppSource](https://appsource.microsoft.com) 发布者指南 本指南旨在帮助新的和现有发布者了解如何利用 Azure Marketplace 与 AppSource 的店面、发布应用程序和服务所要执行的步骤，以及如何在与 Microsoft 的合作中发展自己的业务。 完成本指南后，便可了解有关下列主题的详细信息及在何处查找更多详细信息：

- 加入 Marketplace 店面有什么好处
- 什么是 Azure Marketplace 和 AppSource
- 如何利用这些店面
- 哪种店面最适合所提供的产品和服务 
- 可以发布哪些类型的应用程序和服务产品
- 每个发布选项都有什么技术和商业需求
- 预发布资产清单
- 如何成为发布者
- 在哪里创建和发布产品
- 如何优化列表方式并使用市场推广资源增强影响
- 如果对 Azure Marketplace、AppSource 或本发布指南存在疑问，请在 cloudmarketplace@microsoft.com 联系 Marketplace 团队以获取帮助和支持。 

## <a name="benefits-of-participating-in-the-marketplace"></a>加入 Marketplace 的好处

Azure Marketplace 和 AppSource 是与 Microsoft 联合进行市场推广活动的起点，也是加速业务增长的飞轮。 通过提供推广支持、刺激需求和联合进行市场营销，Marketplace 产品组合可成为云业务引擎的核心。 加入 Marketplace 不收取任何费用。 我们的目标是实现 Microsoft 客户与我们的伙伴生态系统提供的最佳解决方案和服务之间的连接。

充分利用 Marketplace 的功能来扩展业务：

- **生成潜在顾客和销售商机。** 通过 Microsoft 云平台上已扩展的解决方案组合进入新的市场。 对 Marketplace 产品进行追加销售和交叉销售。 
- **提高业务价值并增加与现有客户及新客户的交易规模** 在将工作负载移至云时增加交易规模并解决客户难题。 减少销售周期，通过销售面向特定工作负荷和行业方案的完整解决方案增加交易收益。
- **获取可操作见解。** 你的成功就是我们的成功。 通过云合作伙伴门户洞察产品的性能。 了解执行的内容、所生成的潜在顾客以及最大程度优化市场活动的方式。

>[!NOTE]
>扩展 Office 的应用将通过 Office 应用发布过程访问见解。

## <a name="what-are-azure-marketplace-and-appsource"></a>什么是 Azure Marketplace 和 AppSource？

Microsoft 提供两种不同的 Marketplace 店面，让合作伙伴可以列出产品/服务、提供试用版并与 Microsoft 的客户和生态系统直接交易：[Azure Marketplace (https://azuremarketplace.microsoft.com) 和 [AppSource] (https://appsource.microsoft.com)。 这些店面可以帮助客户查找、试用并购买能加快数字转换的应用程序和服务，并通过增加 Microsoft 客户和合作伙伴的消费者访问量帮助发布者扩展业务。
 
Marketplace 店面与受众需求和 Microsoft 云产品的特点相匹配，从而帮助客户找到所需产品。 每个店面都提供专门的发布选项来帮助你最大程度地优化发布投资，这些投资总结为下表内容：


|          |Azure Marketplace |AppSource  |
|---------|---------|---------|
|目标读者     |IT 专业人员、开发人员（包括 DBA、SecOps、DevOps 等专家角色）    | 业务线决策者（包括采购、制造、计帐等专业角色）      |
|以扩展为目标     |Azure         | Azure、Dynamics 365、Office 365、Power BI 和 Power 应用       |
|解决方案和服务类型     |  基础结构解决方案和专业服务   | 已完成的业务线云应用程序、Office 365 加载项和专业服务        |
|发布选项     |  与我联系、咨询服务产品、试用版、虚拟机、解决方案模板以及托管应用       |  免费试用、立即获取、体验版、与我联系、咨询服务产品      |
|应用内体验可让用户在其应用程序的上下文中访问应用和服务  | Azure 门户 和 CLI         | Office 365、Dynamics 365、Power BI、Office 客户端应用       |

## <a name="leveraging-these-storefronts"></a>利用这些店面

每个店面都服务于独特的客户需求，并能按角色实现目标定向，使你可以基于客户群提供正确的解决方案或服务。

通过 Azure Marketplace 吸引 IT 专业人员和云开发人员来发现、试用并购买 IaaS、SaaS 及 PaaS 解决方案：


|客户需求  |Azure Marketplace |
|---------|---------|
|**需要使用其他云平台功能来解决业务和技术需求**     |  提供一套不断扩充的互补性应用程序和服务产品组合，并且已对其进行优化，可在 Azure 上运行       |
|**发现很难找到适当的应用程序或服务**    |  提供一站式商店，可供发现、试用并购买用于 Azure 的解决方案和服务        |
|**需要对第三方应用程序和服务使用可缩放的部署机制**   | 允许对第三方应用程序和服务创建和配置可缩放的部署        |
|**需要使用新的应用程序与服务来集成和使用现有解决方案**  |   轻松地将第三方应用程序和服务与 Azure 上的现有解决方案集成      |

企业用户可以通过 **AppSource** 查找、试用和获取业务线 SaaS 应用程序和实施服务，以帮助推动业务成果，减少创造价值所需的时间： 


|客户需求  |AppSource  |
|---------|---------|
|**寻找适合目前所用 Microsoft 产品的业务解决方案** | 允许客户使用第三方应用程序和服务来扩展 Microsoft 的云应用程序和技术。       |
|**可以轻松找到适当的解决方案或实施服务。**    |   提供一站式商店，让客户发现、试用和获取应用程序、服务、加载项，等等      |
|**行业特定的业务线应用程序可以解决客户的具体业务难题**   | 提供已完成的端到端行业解决方案，帮助解决跨多个行业的具体要求     |
|**帮助提高生产力、效率和业务见解的应用**    | 为客户服务、人力资源、运营等业务线提供应用        |
| **经验丰富的实施合作伙伴可帮助根据独特情况改编应用** | AppSource 中为基于 Dynamics 365、Power BI、PowerApps 和第三方应用的解决方案提供咨询服务产品目录，以帮助企业用户找到适当的咨询服务设计来实现可预测的结果 |

## <a name="understanding-the-differences-between-storefronts"></a>了解店面之间的差异

要选择店面，首先从确定产品的目标受众开始：Azure Marketplace 面向 IT 专业人员和开发人员，而 AppSource 面向商业用户。 如果解决方案同时面向这两类受众，则只需发布一次就能同时在两种店面中罗列内容。
 
考虑每种店面的其他优势：

|店面优势  |Azure Marketplace  |AppSource   |
|---------|---------|---------|
|**灵活计费**    | 对于虚拟机，采用“即用即付”计费选项，使用 Microsoft 企业协议或 web 直销模型。 定价选项还包含“免费级别”订阅（即产品永久免费），以及“立即试用”订阅：因促销而在有限时间内免费，之后转换为付费订阅。 针对两种计费选项的发布者，还有一个支持选项是“自带许可”激活，在使用 Azure 应用（例如解决方案模板或托管应用）部署虚拟机的方案中，所有已设置的 Azure 资源都直接向客户收费 | AppSource 提供试用体验的预配，但当前不提供支持交易的发布选项；通过这种方式，用户能够利用当前的订购和计费基础结构，且无需进行额外的投资或更改        |
|**推进与其他合作伙伴的连接**     |Azure Marketplace 目前不支持发布者将服务提供商或交付合作伙伴与产品进行链接，但 2018 年会推出此功能         |  在特定的实现方案中可链接独立软件供应商、系统集成商和托管服务提供商，以支持面向新客户的合作销售      |
|**自动化**     |    Azure Marketplace 目前不支持发布者将服务提供商或交付合作伙伴与产品进行链接     | 通过外接程序预配，利用自动化 SaaS，并使用解决方案模板自动实现基于 SaaS 的数据收集和部署方案        |在特定的实现方案中可链接独立软件供应商、系统集成商和托管服务提供商，以支持面向新客户的合作销售
|**多个云类型**     |   通过 Azure Stack 发布公有云和本地解决方案，或发布到包括中国和德国在内的 Azure 政府和区域云      |    AppSource 当前不支持 Azure Stack、Azure 政府或区域云     |
|**向客户进行上下文演示**     |  让用户能够在 Azure 门户中对解决方案进行上下文搜索（虚拟机和解决方案模板）       |  通过 Dynamics 365、Power BI 和 Office 365 等 Microsoft 产品的应用内体验来接触更多的客户    |

## <a name="select-a-publishing-option"></a>选择发布选项

每个店面都支持多个发布选项和产品/服务类型：列表、试用和事务处理。 选择最能代表应用程序和服务详细信息的产品/服务类型。 使用任何发布选项，合作伙伴都能够共享潜在客户。 

本部分逐步讲解这两个店面中提供的选项。 


|**发布选项**  | **产品/服务类型** | **店面**  |
|---------|---------|---------|
|**列表**    |    与我联系、咨询服务     |  Azure Marketplace、AppSource       |
|**试用**   |     免费试用版、SaaS 试用版、互动演示、体验版    |  Azure Marketplace、AppSource       |
|**事务处理**     |   虚拟机、解决方案模板、托管应用程序      |    Azure Marketplace     |

### <a name="list"></a>列表

当试用级别或事务级别的参与不可行时，使用“与我联系”。 此方法的好处是帮助在市场中发布解决方案的发布者立即开始获取潜在顾客，通过进一步维护这些潜在客户产生基础交易，进而扩展业务规模。 但此方式的缺点是，与其他产品/服务类型相比，对客户的吸引力有限。

>[!IMPORTANT]
>“与我联系”不是建议的推广类型，仅当无法建立试用体验时，才应使用此选项。 最好是使用“试用”和“交易”产品来与客户互动。 如果有任何类型的试用体验，则我们的加入过程会根据具体的方案，引导你使用其中一个选项。

如果产品/服务主要由专业服务构成（例如评估、实施和学习班），请使用“咨询服务”产品/服务类型。 产品/服务的范围、持续时间和价格须是固定的，须面向单个客户，且须即时提供和兑现。

### <a name="trial"></a>试用

提供试用体验可以提升客户参与度，从而让解决方案得到更丰富地展示。 使用试用版，客户可以在购买解决方案前先进行体验。 借助面向客户的试用体验，发布者能获得更多的店面促销商机，随着参与客户的增多，可获得越来越多的潜在顾客。
 
所有的试用选项部署在你的试用环境和/或 Azure 订阅中，而非部署在客户的环境或 Azure 订阅中。 免费试用应以客户为导向，不应设置任何附加购买条件，附加配置应尽量少，以便提供最轻松的使用体验。 试用版必须包含免费支持（至少在试用时间内）。 应通过审慎的评估途径发展和监视试用用户，以获取最佳结果。 我们鼓励发布者同时使用 Marketplace 的引导和发布者自己的应用内智能来监视和管理试用用户。

有 4 种典型的试用方案：


|**试用选项**  |**主要优势**  |**适用情况**  |
|---------|---------|---------|
|**免费试用**    |     通过自动转换为付费服务的方法，让客户在购买前试用产品，并为客户启用概念证明，让客户能够与 Microsoft 销售团队合作 |     解决方案是虚拟机或解决方案模板，或（OR）解决方案是 SaaS 产品并且（AND）提供多租户 SaaS 产品，设有首批运行体验，可让单个客户快速体验，如果拥有单个租户但将客户添加为“来宾用户”的话|
**体验版**     |     让客户能在购买前试用产品，并（AND）在预配置的设置中提供解决方案的引导体验 |   解决方案是虚拟机、解决方案模板或单租户的 SaaS 应用，或预配起来很复杂，或（OR）没有将试用版转换为付费产品的方法 |
|**互动演示**    |  让客户无需进行复杂的设置就能体验产品       |    解决方案需要复杂的设置，且难以在试用时间内完成     |


#### <a name="free-trial"></a>免费试用版

如果解决方案或应用提供基于 SaaS 的免费试用版，请使用“免费试用版”。 此选项从感兴趣的客户中发掘高质量的潜在顾客，以此帮助扩展业务。 免费试用版的形式可以是功能有限或者使用时间有限的试用帐户，且应包含激励元素，促使用户快速转换为付费使用。

#### <a name="test-drive"></a>体验版

如果解决方案是在一个或多个虚拟机上通过 IaaS 或 SaaS 应用部署的，请使用“体验版”。 此方法的优势是可实现对虚拟设备或整个解决方案环境的自动预配，一个合作伙伴托管的客户评估解决方案“指导教程”中对此进行了介绍，该指导不向客户收取额外的费用。 客户无需是现有 Azure 客户，这有助于发展更高质量的潜在顾客。

体验版还有一些其他优点：

- Marketplace 中有 27% 的用户搜索将搜索结果限于仅显示体验版的产品/服务 
- 提供体验版的产品/服务比其他不提供体验版的产品/服务要多产生 38% 的潜在顾客 
- 通过 Azure Marketplace 获取的新客户中有 36% 来自于使用过体验版的客户 
- 启用 Microsoft 卖场销售，更好地了解如何通过共同销售售卖产品

#### <a name="interactive-demo"></a>互动演示

通过互动演示，为客户体验引导式产品体验。 此选项的优点是提供试用体验，且无需为复杂解决方案进行复杂预配。 此选项能让客户大致了解解决方案，并让发布者开始收获潜在顾客，增加产品基础交易和扩展业务的机会。 

### <a name="transact"></a>事务处理

在 Azure Marketplace 中，如果将解决方案作为虚拟设备部署至客户的订阅中，请使用“虚拟机”。 虚拟机通过“即用即付”或启用了自带许可的许可模式，支持完全商务化交易。 Microsoft 托管商务交易并代表发布者向客户收费。 发布者可获得的好处是能够利用客户与 Microsoft 之间达成的首选付费关系（包括企业协议）。 

>[!NOTE]
>这时可针对虚拟设备的 Azure 使用情况，而非发布者的软件许可费用，来使用企业协议的货币承诺。

如果解决方案需要在虚拟设备以外实现部署和配置自动化，请使用“Azure 解决方案模板”。 解决方案模板可以自动执行一个或多个虚拟机资源的预配，此外还可以预配网络和存储资源。 解决方案模板可提供在单个虚拟机和整个基于 IaaS 的解决方案环境中实现自动化的优势。 在[此处](https://github.com/MicrosoftDocs/azure-docs)详细了解如何生成解决方案模板。

如果发布者或客户希望将解决方案托管至第三方（例如 SI 或 MSP），请使用 **Azure 托管应用**将虚拟机或整个基于 IaaS 的解决方案部署至客户的订阅。 在此处深入了解如何[生成托管应用](https://docs.microsoft.com/azure/managed-applications/overview)。 有关常见问题的列表，请参阅 [Azure Marketplace 常见问题解答](https://azure.microsoft.com/marketplace/faq/)。

>[!NOTE]
> 托管应用必须可通过 Marketplace 部署。 如果客户沟通是个问题，请注意，可以通过商机分享来与感兴趣的客户交流。

### <a name="azure-certified"></a>Azure 认证

Azure Marketplace 中发布的所有虚拟机都经过“Azure 认证”计划的测试。 该计划可向客户担保你的虚拟机与 Azure 平台和 Marketplace 销售模型兼容，联机映像安全符合性测试可将病毒和恶意软件包含在内，并可以在产品级添加徽章，从而作为经过验证的解决方案，提升对 Microsoft 企业客户的推广效果。

#### <a name="marketplace-commercial-considerations"></a>Marketplace 商业注意事项

加入 Marketplace 不收取任何费用。 使用列表、试用和 BYOL 事务处理选项进行发布时，不存在任何因参与 Marketplace 需支付的收入分成。 有关详细信息，请参阅我们的 [Marketplace 参与政策](https://azure.microsoft.com/support/legal/marketplace/participation-policies/)。

#### <a name="pay-as-you-go-and-bring-your-own-license-billing-options"></a>即用即付和自带许可计费选项

使用“即用即付”事务处理发布选项时，你的基于使用情况的软件许可收入中，你和 Microsoft 各占 80% 和 20% 的收入分成。 单个产品/服务能按“即用即付”和“自带许可”两种计费模型进行定价，并可作为独立 SKU 在产品级别并存。 这可在云合作伙伴门户中的产品/服务中进行配置。

请看以下示例：

如果启用“即用即付”选项：


|你的许可证费用   | 每小时 $1.00        |
|---------|---------|
|Azure 使用费用（D1/1 核）     | 每小时 $0.14  |
|**由 Microsoft 向客户收费**    | **每小时 $1.14**       |

在此方案中，Microsoft 按每小时 $1.14 收取所发布的虚拟机映像的使用费用。


|**Microsoft 收费** |**每小时 $1.14**  |
|---------|---------|
|Microsoft 将许可证费用的 80% 支付给你 | 每小时 $0.80        |
|Microsoft 保留许可证费用的 20%    | 每小时 $0.20        |
|Microsoft 保留 Azure 使用费用     |   每小时 $0.14      |

相反的，如果启用“自带许可”选项：

|你的许可证费用     | 许可证费用由发布者协商确定和收取        |
|---------|---------|
|Azure 使用费用（D1/1 核）    | 每小时 $0.14         |
|**由 Microsoft 向客户收费**     | **每小时 $0.14**        |

使用事务处理发布选项有个很重要的优势，就是 Microsoft 可以在计取 Azure 基础使用费的同时，直接对客户单独计取许可证费用。 在此方案中，Microsoft 代表你进行计费和收费，所以你无需和客户建立交易关系。 这样可以节省时间和资源，将重心放在产品销售而非收费上。

## <a name="using-azure-active-directory-to-enable-trials"></a>使用 Azure Active Directory 启用试用版
Azure Active Directory (AAD) 是一种云标识服务，它使用行业标准协议（OAuth 和 OpenID Connect）通过 Microsoft 工作或学校帐户实现身份验证。 在[此处](https://www.microsoft.com/en-us/cloud-platform/azure-active-directory-features)了解有关 AAD 的详细信息。 

Microsoft 通过 AAD 对所有 Marketplace 用户进行认证，当已验证的用户单击 Marketplace 中的试用列表并重定向到试用环境时，可将用户直接预配到一个试用中，而无需执行其他登录步骤。 [身份验证期间你的应用从 AAD 接收的令牌] (https://docs.microsoft.com/en-us/azure/active-directory/develop/active-directory-token-and-claims#sample-tokens) 包含有用的客户信息，可使用这些信息在应用内创建用户帐户，这样可以自动获得预配体验并提高转换率。 

使用 AAD 为应用或试用启用一键式身份验证：

- 简化从 Marketplace 转换到试用的客户体验 
- 保持“产品内体验”观感，即使是当用户从 Marketplace 重定向到你的域或试用环境中时也是如此
- 降低用户在重定向时放弃的可能性，因为无需其他登录步骤
- 减少因 AAD 用户过多而产生的部署困难

### <a name="certifying-your-azure-active-directory-integration-for-marketplace"></a>为 Marketplace 验证 Azure Active Directory 集成

对于多租户应用程序：

如果要立即支持 AAD

- 在 Azure 门户中注册你的应用程序
- 启用 AAD 中的多租户支持功能，以获取一键式试用体验
- [在此处了解更多信息](https://docs.microsoft.com/en-us/azure/active-directory/develop/active-directory-integrating-applications)

如果你不熟悉联合 SSO 的 AAD

- 在 Azure 门户中注册你的应用程序
- 使用 [OpenID Connect] (https://docs.microsoft.com/en-us/azure/active-directory/develop/active-directory-protocols-openid-connect-code) 或 [OAuth 2.0](https://docs.microsoft.com/en-us/azure/active-directory/develop/active-directory-protocols-oauth-code) 开发 AAD 的 SSO
- 启用 AAD 中的多租户支持功能，以获取一键式试用体验
- [在此处了解更多信息](https://docs.microsoft.com/en-us/azure/active-directory/develop/active-directory-devhowto-appsource-certified)

对于单租户应用程序：

单租户应用程序有多种选项：

使用 Marketplace 帐户 (Microsoft ID) 访问发布者个人资料。 进入云合作伙伴门户后，最后一步就是将开发人员中心帐户与对应的要发布的 Marketplace 发布者个人信息关联。 此操作可在云合作伙伴门户中通过发布者个人资料屏幕底部的按钮完成。

有关如何使用云合作伙伴门户的详细信息，请参阅门户中的[了解详情](https://cloudpartner.azure.com/#Learn)菜单并单击“文档”部分。

## <a name="guidelines-and-how-tos"></a>指南及操作方法

### <a name="guidelines-for-creating-a-microsoft-id-to-manage-an-azure-marketplace-account"></a>关于创建 Microsoft ID 以管理 Azure Marketplace 帐户的指南

创建公司帐户时，如果多个用户将需要通过使用建立该帐户的 Microsoft 帐户登录来访问该帐户，请遵循以下准则。

>[!IMPORTANT]
>要允许多个用户访问开发人员中心帐户，我们建议使用 Azure Active Directory 为单个用户分配角色，这些用户可以通过使用其个人 Azure AD 凭据登录来访问该帐户。 有关详细信息，请查看 [AAD 联合域指导](#guidance-with-aad-federated-domains)。使用属于企业域（而不是属于单个人）的电子邮件地址（例如，windowsapps@fabrikam.com）创建 Microsoft 帐户。

- 限制只有尽可能最少数量的开发人员可以访问此 Microsoft 帐户。
- 设置公司电子邮件通讯组列表，其中包括需要访问开发人员帐户的所有人员，并将此电子邮件地址添加到安全信息。 在需要的时候，这会允许列表中的所有员工接收安全代码以及管理 Microsoft 帐户的安全信息。 如果设置通讯组列表不可行，则将需要个人电子邮件帐户的所有者可供访问，并在出现提示时（例如，将新的安全信息添加到该帐户时或必须从新设备访问该帐户时）共享安全代码。
- 添加不需要分机且主要团队成员可以访问的公司电话号码。
- 一般情况下，让开发人员使用受信任的设备登录公司的开发人员帐户。 所有主要团队成员应有权访问这些受信任的设备。 这将减少访问帐户时要发送安全代码的需要。
- 如果需要允许从不受信任的电脑访问该帐户，则限制最多只有五个开发人员可以进行该访问。 理想情况下，这些开发人员应从共用同一地理位置和网络位置的计算机访问该帐户。
- 经常在 [https://account.live.com/proofs/Manage](https://account.live.com/proofs/Manage) 中查看公司的安全信息以确保这些信息都是最新的。

应主要从受信任的电脑访问开发人员帐户。 这很关键，因为对每个帐户每周生成的代码数有限制。 它还确保提供最无缝的登录体验。
有关其他开发人员帐户准则和安全性的详细信息，请单击[此处](https://docs.microsoft.com/en-us/windows/uwp/publish/opening-a-developer-account)。

### <a name="guidance-for-microsoft-ids-in-aad-federated-domain"></a>关于 AAD 联盟域中 Microsoft ID 的指南

可使用 [Azure Active Directory](https://docs.microsoft.com/en-us/azure/active-directory/) 联合公司帐户，如果尝试用公司电子邮件地址创建 Microsoft ID，会返回一个错误信息。 如果收到错误信息，请首先与 IT 团队核实导致错误的原因是否是个问题。 这是一个已知问题，我们正在努力解决。 解决办法如下：

建议在 @outlook.com 域中创建新的电子邮件地址。 执行以下步骤:

1. 转到 [https://signup.live.com/signup](https://signup.live.com/signup)，并选择“获取新的电子邮件地址”


2. 创建新电子邮件地址，并输入密码。 这将在 outlook.com 服务中同时创建新 Microsoft ID 和电子邮箱。 继续执行注册流程，直到帐户创建完毕。

>[!IMPORTANT]
>确保用于在开发人员中心中注册的电子邮件 ID 或通讯组列表（建议使用通讯组列表，以删除个人的依赖性）是首次注册为 Microsoft 帐户。 如果不是，请使用此[链接](https://signup.live.com/signup?uaid=e479342fe2824efeb0c3d92c8f961fd3&lic=1)进行注册。

此外，Microsoft 公司域下的任何电子邮件 ID 都不能用于开发人员中心注册。

使用这个帐号创建了 Microsoft ID 以后，登录帐户[邮箱](https://outlook.live.com/owa/)，并创建电子邮件转发规则，将此邮箱中的所有电子邮件移动至域中创建的电子邮件地址，用于管理 Azure Marketplace 帐户。 要了解如何在 outlook.com 中创建电子邮件转发规则，请参阅此[链接](https://support.office.com/en-us/article/Use-rules-in-Outlook-Web-App-to-automatically-forward-messages-to-another-account-1433e3a0-7fb0-4999-b536-50e05cb67fed)。

完成上一步骤后，即将所有来自 Microsoft ID 的电子邮件/讯息都发送到域中的电子邮件帐户。 需要使用 @outlook.com 电子邮件地址在 Microsoft 开发人员中心和云合作伙伴门户中进行身份验证。

### <a name="instructions-on-how-to-register-in-the-development-center"></a>关于如何在开发中心中注册的说明

1. 打开新的 Internet Explorer InPrivate 或 Chrome Incognito 浏览会话，以确保未登录到个人帐户。
2. 在开发人员中心中转到 [http://dev.windows.com/registration?accountprogram=azure](http://dev.windows.com/registration?accountprogram=azure) 将自己注册为卖家。 请在继续操作之前，阅读以下重要说明。

   ![开发人员中心电子邮件](./media/marketplace-publishers-guide/registerdevcenteremail.png)

3. 完成“帮助我们保护帐户”向导，该向导会通过电话号码或电子邮件地址验证身份。
4. 在“注册帐户信息”部分中，请从下拉菜单中选择“帐户所在国家/地区”。

   ![帐户信息](./media/marketplace-publishers-guide/devcenterregistrationaccountinfo.png)
   
   >[!WARNING]
   >“销售”国家/地区：若要在 Microsoft Azure Marketplace 销售服务，注册实体必须属于上述已批准的“销售”国家/地区之一。 此限制是出于付款和税务的原因。 有关详细信息，请参阅 [Marketplace 参与策略]https://azure.microsoft.com/support/legal/marketplace/participation-policies/。

5. 将“帐户类型”选择为“公司”，并单击“下一步”按钮。

   >[!IMPORTANT]
   >要进一步了解帐户类型以及哪一个是最佳选择，请查看[帐户类型、位置和费用](https://docs.microsoft.com/en-us/windows/uwp/publish/account-types-locations-and-fees)页。

6. 输入“发布者显示名称”，通常为公司名称。

   >[!TIP]
   >优惠列出后，开发人员中心中输入的发布者显示名称不会显示在 Azure Marketplace 中。 但若要完成注册过程，必须填写该名称。

7. 输入用于帐户验证的“联系人信息”。

   >[!IMPORTANT]
   >必须提供准确的联系人信息，因为该信息用于对公司在开发人员中心中获得批准的验证过程。

8. 输入“公司审批者”的联系人信息。 公司审批者是如下人员：可以验证你是否有权代表组织在开发人员中心中创建帐户。 完成后，单击“下一步”以转到“付款部分”。

   ![开发人员中心支付](./media/marketplace-publishers-guide/devcenterregistrationpayment.png)

9. 输入付款信息，以为帐户付款。 如果有抵扣注册费用的促销代码，可以在此处输入该代码。 否则，请提供信用卡信息，或所在市场支持的 PayPal。 完成后，单击“下一步”以转到“查看屏幕”

   ![开发人员中心支付](./media/marketplace-publishers-guide/devcenterregistrationpayment2.png)

10. 查看帐户信息，并确认所有内容都正确无误。 然后，阅读并接受“Microsoft Azure Marketplace 发布者协议”的条款和条件。 选中该框，即表明已阅读并接受这些条款。
11. 单击“完成”，确认注册。 我们会向电子邮件地址发送一封确认邮件。
12. 如果打算只发布免费的产品/服务，请单击“转到[云合作伙伴门户](https://cloudpartner.azure.com)”。

如果打算发布商业（事务处理）产品/服务（例如，带有每小时计费模型的虚拟机产品），请单击“更新帐户信息”，其中必须在开发人员中心帐户中填写税务和银行信息。

如果希望稍后更新税务和银行信息，则可以转到下一个部分。 

>[!IMPORTANT]
>如果是商业（事务处理）产品/服务，在未填写税务和银行帐户信息的情况下，无法将产品/服务投入生产。

#### <a name="add-tax-and-banking-information"></a>添加税务和银行信息

如果想要发布供购买的商业产品/服务，还需要在开发人员中心添加付款和税务信息，并提交该信息进行验证。 如果只发布免费产品/服务或 BYOL 产品/服务，则不需要添加此类信息。 可以以后添加上述信息，不过验证税务信息需要一些时间。 如果知道会提供供购买的商业产品/服务，建议尽快添加上述信息。

**银行信息**
1. 使用 Microsoft 帐户登录到 Microsoft 开发人员中心。
2.  单击左侧菜单中的“付款帐户”，在“选择付款方式”下单击“银行帐户”或“PayPal”。

   >    [!IMPORTANT]
   >    如果拥有客户在 Marketplace 购买过的商业产品/服务，该帐户即用于对这些购买进行收款。

3. 输入付款信息，确认无误后单击“保存”。

   >    [!IMPORTANT]
   >    如果需要更新或更改付款帐户，请按照上述相同步骤操作，以将当前信息替换为新信息。 更改付款帐户可能会推迟付款（最多一个付款周期）。 出现此推迟的原因是我们需要验证帐户更改，正如首次设置付款帐户时我们会进行验证。 帐户通过验证后，仍会收到全额款项；当前付款周期应得的任何付款将添加到下一个付款周期。

4. 单击“下一步” 

**税务信息**

1. 必要时使用 Microsoft 帐户登录到 [Microsoft 开发人员中心](https://dev.windows.com)。
2. 单击左侧菜单中的“税务资料”。
3. 在“设置纳税表单”页上，选择具有永久居留权的国家或地区，并选择保留原籍的国家或地区。 单击“资源组名称” 的 Azure 数据工厂。
4. 输入税务详细信息，然后单击“下一步”

   >    [!WARNING]
   >    在未在 Microsoft 开发人员中心帐户中填写税务和银行帐户信息的情况下，将无法投入生成商业产品/服务。

如果在进行开发人员中心注册时遇到问题，请记录支持票证，如下所示：

1. 请转到支持链接 https://developer.microsoft.com/windows/support
2. 在“联系我们”部分下，单击“提交事件”按钮，如以下屏幕截图中所示。
3. 选择“开发人员中心帮助”作为“问题类型”，选择“发布和管理应用”作为“类别”。 完成后，单击“启动电子邮件”按钮。

   ![开发人员中心问题](./media/marketplace-publishers-guide/devcentersubmitincident.png)

4. 系统将提供登录页面。 使用任何 Microsoft 帐户登录。 如果没有 Microsoft 帐户，可以使用此链接创建一个帐户。 
5. 填写问题的详细信息，并通过单击“提交”按钮来提交该票证。

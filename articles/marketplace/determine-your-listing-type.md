# <a name="determine-the-listing-type-right-for-your-offer"></a>确定适合你的产品/服务的列表类型

每个店面都支持多个发布选项和产品/服务类型。 选择最能代表应用程序和服务详细信息的产品/服务类型。 使用任何发布选项，合作伙伴都能够共享潜在客户。 


## <a name="list"></a>列出

当试用级别或事务级别的参与不可行时，使用“与我联系”。 此方法的好处是帮助在市场中发布解决方案的发布者立即开始获取潜在顾客，通过进一步维护这些潜在客户产生基础交易，进而扩展业务规模。 但此方式的缺点是，与其他产品/服务类型相比，对客户的吸引力有限。

>[!IMPORTANT]
>最好是使用“试用”和“交易”产品来与客户互动。 “与我联系”的价值是你收到的潜在顾客，因此如果你选择此列表类型，请确保已配置潜在顾客目标，并且已准备好使潜在顾客数最大化。 

如果产品/服务主要由专业服务构成（例如评估、实施和学习班），请使用“咨询服务”产品/服务类型。 产品/服务的范围、持续时间和价格须是固定的，须面向单个客户，且须即时提供和兑现。

## <a name="trial"></a>试用

提供试用体验可以提升客户参与度，从而让解决方案得到更丰富地展示。 使用试用版，客户可以在购买解决方案前先进行体验。 借助面向客户的试用体验，发布者能获得更多的店面促销商机，随着参与客户的增多，可获得越来越多的潜在顾客。
 
所有的试用选项部署在你的试用环境和/或 Azure 订阅中，而非部署在客户的环境或 Azure 订阅中。 免费试用应以客户为导向，不应设置任何附加购买条件，附加配置应尽量少，以便提供最轻松的使用体验。 试用版必须包含免费支持（至少在试用时间内）。 应通过审慎的评估途径发展和监视试用用户，以获取最佳结果。 我们鼓励发布者同时使用 Marketplace 的引导和发布者自己的应用内智能来监视和管理试用用户。

有三种典型的试用场景：

**免费试用**

如果解决方案或应用提供基于 SaaS 的免费试用版，请使用免费试用。 此选项从感兴趣的客户中发掘高质量的潜在顾客，以此帮助扩展业务。 免费试用的形式可以是功能有限或者使用时间有限的试用帐户。 它们应包含激励元素，促使用户快速转换为付费使用。

|**试用选项**  |**主要优势**  |**适用情况**  |
|---------|---------|---------|
|**免费试用**    |     通过自动转换为付费服务的方法，让客户在购买前试用产品。 此外，为客户启用概念证明，让客户能够与 Microsoft 销售团队合作。 |     解决方案是虚拟机或解决方案模板。<br><br> 解决方案是 SaaS 产品/服务并提供多租户 SaaS 产品。 <br><br>设有首批运行体验，可让单个客户快速体验。 <br><br>有单个租户但将客户添加为来宾用户。|

**体验版**

如果解决方案是在一个或多个虚拟机上通过 IaaS 或 SaaS 应用部署的，请使用体验版。 此方法的优势是可实现对虚拟设备或整个解决方案环境的自动预配，一个合作伙伴托管的客户评估解决方案“指导教程”中对此进行了介绍，该指导不向客户收取额外的费用。 客户无需是现有 Azure 客户，这有助于发展更高质量的潜在顾客。

体验版还有一些其他优点：

- Marketplace 中有 27% 的用户搜索将搜索结果限于仅显示体验版的产品/服务。 
- 提供体验版的产品/服务比其他不提供体验版的产品/服务要多产生 38% 的潜在顾客。 
- 通过 Marketplace 获取的新客户中有 36% 来自于使用过体验版的客户 
- 体验版启用 Microsoft 卖场销售，更好地了解如何通过共同销售售卖产品。

|**试用选项**  |**主要优势**  |**适用情况**  |
|---------|---------|---------|
|**体验版**     |     让客户能在购买前试用产品。 此外，在预配置的设置中提供解决方案的引导体验。 |   解决方案是虚拟机、解决方案模板或单租户的 SaaS 应用，或预配起来很复杂。 <br><br>没有将试用版转换为付费产品/服务的方法。 |

**互动演示**

通过互动演示，为客户体验引导式产品体验。 此选项的优点是提供试用体验，且无需为复杂解决方案进行复杂预配。 此选项能让客户大致了解解决方案。 此外，它可让发布者开始收获潜在顾客，增加产品基础交易和扩展业务的机会。 

|**试用选项**  |**主要优势**  |**适用情况**  |
|---------|---------|---------|
|**互动演示**    |  让客户无需进行复杂的设置就能体验产品。       |    解决方案需要复杂的设置，且难以在试用时间内完成。     |

## <a name="transaction"></a>事务

在 Azure Marketplace 中，如果将解决方案作为虚拟设备部署至客户的订阅中，请使用虚拟机。 虚拟机通过“即用即付”或启用了自带许可的许可模式，支持完全商务化交易。 Microsoft 托管商务交易并代表发布者向客户收费。 发布者可获得的好处是能够利用客户与 Microsoft 之间达成的首选付费关系（包括企业协议）。 

>[!NOTE]
>目前可针对虚拟设备的 Azure 使用情况，而非发布者的软件许可费用，来使用企业协议的货币承诺。

>[!NOTE]
>通过将映像和定价列为专有产品/服务，可以将虚拟机的发现和部署限制为特定的一组客户。 专有产品/服务让发布者能够为关系最密切的客户提供专属优惠，并为他们提供自定义软件和条款。 这些自定义条款可在一系列场景中为发布者提供助力，包括使用特殊定价和条款的现场导向型交易以及对限制性版本软件的早期访问。 这让发布者可以通过创建包含这些细节的新 SKU，向有限的一组客户提供特定的定价或产品。 详细了解[专有产品/服务](https://azure.microsoft.com/en-us/blog/private-offers-on-azure-marketplace/)。

如果解决方案需要在虚拟设备以外实现部署和配置自动化，请使用 Azure 应用程序解决方案模板。 解决方案模板可以自动执行一个或多个虚拟机资源的预配，此外还可以预配网络和存储资源。 解决方案模板可提供在单个虚拟机和整个基于 IaaS 的解决方案环境中实现自动化的优势。 在 [GitHub](https://github.com/MicrosoftDocs/azure-docs) 中详细了解如何生成解决方案模板。

如果发布者或客户希望将解决方案托管至第三方（例如 SI 或 MSP），请使用 Azure 托管应用将虚拟机或整个基于 IaaS 的解决方案部署至客户的订阅。 在 [Azure 托管应用程序概述](https://docs.microsoft.com/azure/managed-applications/overview)中深入了解如何生成托管应用。 有关常见问题的列表，请参阅 [Marketplace 常见问题解答](https://azure.microsoft.com/marketplace/faq/)。

>[!NOTE]
> 托管应用必须可通过 Marketplace 部署。 如果客户沟通是个问题，请注意，可以通过商机分享来与感兴趣的客户交流。

对于基于 SaaS 的技术解决方案，请使用“SaaS 应用”产品/服务类型，使 Azure 客户可以通过 Azure Marketplace 购买解决方案的订阅。 若要使用此功能，发布者必须按照每月固定费率对其服务进行定价和计费，并提供随时升级或取消服务的功能。 与虚拟机类似，Microsoft 托管商务交易并代表发布者向客户收费。 若要将 SaaS 应用用作订阅，发布者需启用自己的订阅管理服务 API，直接与 Azure Marketplace 预配 API 进行通信，以支持从 Azure 门户进行服务预配、升级和取消。

使用“容器”将解决方案作为要预配的 Docker 容器映像发布到基于 Kubernetes 的 Azure 容器服务，比如 Azure Kubernetes 服务或 Azure 容器实例（基于 Kubernetes 的容器运行时的用户选择）。 Microsoft 目前支持免费的和已启用 BYOL 的许可模式，并且在未来几个月内将推出更多商业功能，包括订阅和基于使用的计费选项。

## <a name="azure-certified-program"></a>Azure 认证计划

Azure Marketplace 中发布的所有虚拟机都经过“Azure 认证”计划的测试。 该计划：

- 可向客户担保你的虚拟机与 Azure 平台和 Marketplace 销售模型兼容。
- 联机映像安全符合性测试可将病毒和恶意软件包含在内。
- 可以在产品/服务级添加徽章，从而作为经过验证的解决方案，提升对 Microsoft 企业客户的推广效果。
- 有关详细信息，请查看 [Microsoft Azure 认证](https://azure.microsoft.com/en-us/marketplace/programs/certified/)。

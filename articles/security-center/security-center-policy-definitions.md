# <a name="azure-security-policies-monitored-by-azure-security-center"></a>通过 Azure 安全中心监视的 Azure 安全策略
本文提供了可在安全中心监视的 Azure Policy 定义的列表。

## <a name="available-security-policies"></a>可用的安全策略

若要了解通过安全中心监视的内置策略，请参阅下表：

| 策略 | 策略的用途 |
| --- | --- |
|审核确认已在 Service Fabric 和虚拟机规模集中启用诊断日志|建议启用日志，以便在出现某个事件或遭到入侵后需要进行调查时可以重新创建活动线索。|
|审核事件中心命名空间的授权规则|服务中心客户端不应使用提供对命名空间中所有队列和主题的访问的命名空间级访问策略。 若要符合最低特权安全模型，应在实体级别针对队列和主题创建访问策略，以便仅提供对特定实体的访问。|
|审核确认对事件中心实体使用了授权规则|审核确认对事件中心实体使用了授权规则，用于授予最低访问权限。|
|审核对存储帐户的不受限的网络访问|在存储帐户防火墙设置中审核无限制的网络访问权限。 应该配置网络规则，以便只有来自许可网络的应用程序才能访问存储帐户。 若要允许特定的 Internet 或本地客户端建立连接，可以授予对来自特定 Azure 虚拟网络的流量或者对公共 Internet IP 地址范围的访问权限。|
|审核自定义 RBAC 规则的使用|审核“所有者、参与者、读者”等内置角色而不是容易出错的自定义 RBAC 角色。 使用自定义角色会被视作异常，需要严格的审查和威胁建模。|
|审核确认已在 Azure 流分析中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。|
|审核确认指向存储帐户的传输的安全性|审核存储帐户中安全传输的要求。 安全传输选项会强制存储帐户仅接受来自安全连接 (HTTPS) 的请求。 使用 HTTPS 可确保服务器和服务之间的身份验证并保护传输中的数据免受中间人攻击、窃听攻击和会话劫持等网络层攻击。|
|审核确认已为 SQL Server 预配了 Azure Active Directory 管理员|审核确认已为 SQL Server 预配了 Azure Active Directory 管理员以启用 Azure AD 身份验证。 使用 Azure AD 身份验证可以简化权限管理，以及集中化数据库用户和其他 Microsoft 服务的标识管理。|
|审核服务总线命名空间的授权规则|服务总线客户端不应使用提供对命名空间中所有队列和主题的访问的命名空间级访问策略。 若要符合最低特权安全模型，应在实体级别针对队列和主题创建访问策略，以便仅提供对特定实体的访问。|
|审核确认已在服务总线中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。|
|审核确认 Service Fabric 中的 ClusterProtectionLevel 属性设置为 EncryptAndSign|Service Fabric 使用主要群集证书为节点之间的通信提供三个保护级别（None、Sign 和 EncryptAndSign）。 请设置保护级别，确保节点到节点的所有消息经过加密和数字签名。| 
|审核确认已在 Service Fabric 中使用 Azure Active Directory，用于实施客户端身份验证|审核 Service Fabric 中仅通过 Azure Active Directory 进行客户端身份验证| 
|审核确认已启用搜索服务诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核是否仅启用与 Azure Redis 缓存的安全连接|审核是否仅启用通过 SSL 来与 Azure Redis 缓存建立连接。 使用安全连接可确保服务器和服务之间的身份验证并保护传输中的数据免受中间人攻击、窃听攻击和会话劫持等网络层攻击| 
|审核确认已在逻辑应用中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核确认已在 Key Vault 中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。|
|审核确认已在事件中心启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核确认已在 Azure Data Lake Store 中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核确认已在 Data Lake Analytics 中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核经典存储帐户的使用|为存储帐户使用 Azure 资源管理器以提供安全增强功能，例如：更强的访问控制 (RBAC)、更好地审核、基于 Azure 资源管理器的部署和治理、托管标识访问权限、用于提供机密的 Key Vault 的访问权限、基于 Azure AD 的身份验证以及可实现更轻松安全管理的标记和资源组支持。| 
|审核经典虚拟机的使用|为虚拟机使用 Azure 资源管理器以提供安全增强功能，例如：更强的访问控制 (RBAC)、更好地审核、基于 Azure 资源管理器的部署和治理、托管标识访问权限、用于提供机密的 Key Vault 的访问权限、基于 Azure AD 的身份验证以及可实现更轻松安全管理的标记和资源组支持。| 
|审核确认已配置 Batch 帐户指标警报规则|审核确认已配置 Batch 帐户指标警报规则，以启用所需指标。| 
|审核确认已配置 Batch 帐户指标警报规则|审核确认已配置 Batch 帐户指标警报规则，以启用所需指标。| 
|审核确认已在 Batch 帐户中启用诊断日志|审核确认已启用日志并将日志保留长达一年。 这样便可以在发生安全事件或网络遭泄露时，重新创建活动线索用于调查目的。| 
|审核确认已启用自动化帐户变量加密功能|存储敏感数据时，请务必启用自动化帐户变量资产加密。| 
|审核确认已在应用服务中启用诊断日志|审核确认已在应用上启用诊断日志。 如果发生安全事件或网络遭泄露，这样便可以重新创建活动线索用于调查目的。| 
|审核透明数据加密状态|审核 SQL 数据库的透明数据加密状态。| 
|审核 SQL 服务器级别审核设置|在服务器级别审核 SQL 审核的存在。| 
|[预览]:监视 Azure 安全中心内未加密的 SQL 数据库|建议通过 Azure 安全中心监视未加密的 SQL Server 或数据库。| 
|[预览]:监视 Azure 安全中心内未审核的 SQL 数据库|建议通过 Azure 安全中心监视未启用 SQL 审核的 SQL Server 和数据库。| 
|[预览]:监视 Azure 安全中心内系统更新的缺失情况|建议通过 Azure 安全中心监视服务器上缺失的安全系统更新。| 
|[预览]:审核存储帐户是否缺少 blob 加密|此策略审核没有 blob 加密的存储帐户。 仅适用于 Microsoft.Storage 资源类型，不适用于其他存储提供程序。 建议通过 Azure 安全中心监视可能的网络即时访问。| 
|[预览]:监视 Azure 安全中心内可能的网络即时(JIT)访问|建议通过 Azure 安全中心监视可能的网络即时(JIT)访问。| 
|[预览]:监视 Azure 安全中心内列入允许列表的可能的应用|通过 Azure 安全中心监视可能的应用程序允许列表配置。| 
|[预览]:监视 Azure 安全中心内规则较宽松的网络访问|建议通过 Azure 安全中心监视规则过于宽松的网络安全组。| 
|[预览]:监视 Azure 安全中心的 OS 漏洞|建议通过 Azure 安全中心监视不满足配置的基线的服务器。| 
|[预览]:监视 Azure 安全中心 Endpoint Protection 的缺失情况|建议通过 Azure 安全中心监视未安装 Endpoint Protection 代理的服务器。| 
|[预览]:监视 Azure 安全中心内未加密的 VM 磁盘|建议通过 Azure 安全中心监视未启用磁盘加密的 VM。| 
|[预览]:监视 Azure 安全中心的 VM 漏洞|建议在 Azure 安全中心监视漏洞评估解决方案检测到的漏洞和没有漏洞评估解决方案的 VM。| 
|[预览]:监视 Azure 安全中心内未受保护的 Web 应用程序|建议通过 Azure 安全中心监视未受 Web 应用程序防火墙保护的 Web 应用程序。| 
|[预览]:监视 Azure 安全中心内未受保护的网络终结点|建议通过 Azure 安全中心监视未受下一代防火墙保护的网络终结点。| 
|[预览]:监视 Azure 安全中心的 SQL 漏洞评估结果|监视漏洞评估扫描结果并提供如何补救数据库漏洞的相关建议。| 
|[预览]:审核最大订阅所有者数|建议最多指定 3 个订阅所有者，以减少可能出现的已遭入侵的所有者做出的违规行为。| 
|[预览]:审核最大订阅所有者数|建议指定多个订阅所有者，这样才会有管理员访问冗余。| 
|[预览]:审核具有所有者权限但未启用 MFA 的订阅帐户|为了防止帐户或资源出现违规问题，应为所有拥有所有者权限的订阅帐户启用多重身份验证 (MFA)。| 
|[预览]:审核具有写入权限但未启用 MFA 的订阅帐户|为了防止帐户或资源出现违规问题，应为所有拥有写入特权的订阅帐户启用多重身份验证 (MFA)。| 
|[预览]:审核具有读取权限但未启用 MFA 的订阅帐户|为了防止帐户或资源出现违规问题，应为所有拥有读取特权的订阅帐户启用多重身份验证 (MFA)。| 
|[预览]:审核具有所有者权限但已被弃用的订阅帐户|应从订阅中删除拥有所有者权限的已弃用帐户。 已弃用帐户是已阻止登录的帐户。| 
|[预览]:审核已弃用的订阅帐户|应从订阅中删除弃用的帐户。 已弃用帐户是已阻止登录的帐户。| 
|[预览]:审核具有所有者权限的外部订阅帐户|为了防止发生未受监视的访问，应从订阅中删除拥有所有者权限的外部帐户。| 
|[预览]:审核具有写入权限的外部订阅帐户|应从订阅中删除拥有写入特权的外部帐户，以防发生未受监视的访问。| 
|[预览]:审核具有读取权限的外部订阅帐户|应从订阅中删除拥有读取特权的外部帐户，以防发生未受监视的访问。|

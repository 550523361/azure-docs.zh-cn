# <a name="terminology"></a>术语

个性化器使用强化学习中的术语。 这些术语在 Azure 门户和 API 中使用。

## <a name="conceptual-terminology"></a>概念术语

* **学习循环**：您为应用程序的每个部分创建一个个性化资源，称为_学习循环_，这些部分可以从个性化中受益。 如果需要个性化多个体验，请为每个体验创建一个循环。

* **模型**：个性化模型捕获有关用户行为的所有数据，从发送到 Rank 和 Reward 调用的参数的组合以及学习策略确定的培训行为中获取训练数据。

## <a name="personalizer-configuration"></a>个性化配置

个性化器是从 Azure[门户](https://portal.azure.com)配置的。

* **奖励**：配置奖励等待时间、默认奖励和奖励聚合策略的默认值。

* **探索**：配置用于勘探的排名调用的百分比

* **模型更新频率**：重新训练模型的频率。

* **数据保留**：要存储的数据价值多少天。 这可能会影响脱机评估，这些评估用于改善学习循环。

## <a name="use-rank-and-reward-apis"></a>使用排名和奖励 API

* **排名**： 给定具有功能和上下文功能的操作，请使用浏览或利用来返回顶部操作（内容项）。

    * **操作**：操作是要选择的内容项，如产品或促销。 个性化者选择顶级操作（返回的奖励操作 ID）通过排名 API 向用户显示。

    * **上下文**：要提供更准确的排名，请提供有关上下文的信息，例如：
        * 你的用户。
        * 用户所用的设备。
        * 当前时间。
        * 有关当前情况的其他数据。
        * 有关用户或上下文的历史数据。

        特定的应用程序可能具有不同的上下文信息。

    * **[功能](concepts-features.md)**：有关内容项或用户上下文的信息单元。 请确保仅使用聚合的功能。 请勿将特定时间、用户专用时或其他非聚合数据用作功能。

        * _操作功能_是关于内容的元数据。
        * _上下文功能_是关于显示内容的上下文的元数据。

* **探索**：个性化服务正在探索何时，而不是返回最佳操作，而是为用户选择不同的操作。 个性化体验创建服务可以避免偏差、停滞，并可通过探索来适应现行的用户行为。

* **利用**： 个性化服务使用当前模型根据过去的数据决定最佳操作。

* **实验持续时间**：从为该事件进行排名调用的那一刻起，个性化服务等待奖励的时间量。

* **非活动事件**：非活动事件是您调用 Rank 的事件，但您不确定用户是否会看到结果，因为客户端应用程序决策。 使用非活动事件可以创建和存储个性化结果，然后决定放弃这些结果，而不会影响机器学习模型。


* **奖励**：用户如何响应 Rank API 返回的奖励操作 ID 的度量，分数介于 0 到 1 之间。 0 到 1 的值由业务逻辑设置，依据是所做的选择对实现个性化业务目标提供了多大的帮助。 学习循环不会将此奖励存储为单个用户历史记录。

## <a name="offline-evaluations"></a>脱机评估

* **评估**：脱机评估根据循环的数据确定循环的最佳学习策略。

* **学习策略**：个性化器如何在每个事件上训练模型将由影响机器学习算法工作方式的一些参数决定。 新的学习循环以默认**的学习策略**开头，它可以产生中等的性能。 运行[评估](concepts-offline-evaluation.md)时，个性化程序会创建新的学习策略，专门针对循环的用例进行优化。 使用针对每个特定循环优化的策略（在评估期间生成）个性化程序的性能将显著提高。 学习策略在 Azure 门户中的"**个性化"资源模型和学习设置**上命名为_学习设置_。

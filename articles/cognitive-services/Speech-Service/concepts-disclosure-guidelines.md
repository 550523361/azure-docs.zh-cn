# <a name="disclosure-design-guidelines"></a>披露设计准则
了解如何通过对你的语音体验的综合性质透明，来构建和维护与客户的信任。

## <a name="what-is-disclosure"></a>什么是公开？

泄漏是让人们知道他们与 synthetically 生成&#39;的语音进行交互或倾听其相互交流的一种方法。

## <a name="why-is-disclosure-necessary"></a>为什么需要公开？

公开计算机生成的语音的合成源的需要相对较新。 过去，计算机生成的语音显然很明显，任何人都不会对真实人员产生任何错误。 不过，这种情况下，综合语音的真实性会提高，并且与人为语音相比，它们变得越来越难以区分。

## <a name="goals"></a>目标
在设计综合语音体验时，需要牢记以下原则：

**强化信任**
<br>设计使把测试失败，而不会降低体验。 让用户注意到他们正在与综合语音交互，同时允许他们无缝地利用体验。

**适应使用上下文**
<br>了解你的用户将如何与综合语音交互，在正确的时间提供正确类型的公开。

**设置明确期望**
<br>让用户能够轻松发现和理解代理功能。 提供在请求时了解有关综合语音技术的详细信息。

**接纳失败**
<br>使用失败时间来强化代理功能。

## <a name="how-to-use-this-guide"></a>如何使用本指南

本指南可帮助你确定哪些披露模式最适合你的综合语音体验。 接下来，我们将提供有关如何以及何时使用它们的示例。 其中的每种模式旨在使用户能够最大程度地提高合成语音的透明度，同时保持对人为中心的设计。

考虑到有关语音体验的丰富设计指南，我们将重点放在下面：

1. [**公开评估**](#disclosure-assessment)：确定建议用于综合语音体验的泄露类型的过程

2. [**如何披露**](concepts-disclosure-patterns.md)：可应用于综合语音体验的公开模式示例

3. [**何时公开**](concepts-disclosure-patterns.md#when-to-disclose)：在整个用户旅程中获得最佳时间

## <a name="disclosure-assessment"></a>泄露评估
请考虑您&#39;的用户对交互的期望，并考虑他们在其中体验语音的上下文。 如果上下文清楚地表明合成语音 &quot;说，&quot; 泄露可能会很少、被暂时甚至不必要。 影响公开的上下文的主要类型包括角色类型、方案类型和公开程度。 它还有助于考虑可能正在侦听的人。

### <a name="understand-context"></a>了解上下文

使用此工作表来确定综合语音体验的上下文。 你将在下一步中应用此项，你将在其中确定你的公开级别。

|                                    | 使用上下文                                                                                                                                                                                                                                                                                                                                                       | 潜在风险 & 挑战                                                                                                                                                                                                                                                                                                                                                                       |
|------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **1. 角色类型**               | **如果满足以下任一条件，则你的角色适合于 "用户赞的角色" 类别：**<br><br><ul><li> 角色只是一种真实的工作方式。 （例如，照片或计算机生成的真实人员渲染）<br><br><li> 综合语音以可识别真实人员（例如名人、政治图）的语音为基础 | 你为你的角色提供了更有人力的表示形式，用户将与真实人员关联起来就越有可能，或导致他们认为内容是由真实人员说出的，而不是由计算机生成的。 </ul>                                                                                                                                                                      |
| **2. 方案类型**            | **如果满足以下任一条件，则语音体验符合 "敏感" 类别：**<br><br><ul><li> 获取或显示用户的个人信息 <br><br> <li> 广播时间敏感新闻/信息（例如紧急警报）<br><br><li> 旨在帮助真实人员相互通信（例如，读取个人电子邮件/短信）<br><br> <li> 提供医疗/卫生援助 </ul>            | 当主题与敏感、个人或紧急问题相关时，使用综合语音可能并不合适或不值得信任。 他们还可能期望与真实人员理解相同级别的理解和上下文认知。 |
| **3. 公开级别** |**如果出现以下情况，你的语音体验最可能符合 "高" 类别：** <br><br><ul><li>用户会经常听到或与综合语音交互，或者长时间 </ul>                                                                                                                                                                             | 在建立长期关系时，透明并与用户建立信任的重要性更高。                                                                                                                                                                                                                                                                      |

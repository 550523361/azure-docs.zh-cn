# <a name="define-your-mobile-engagement-strategy"></a>定义 Mobile Engagement 策略
*编写应用是为了让用户用它！*

可以确信的是，为了让用户喜欢，需要花费很大力气编写应用。 此外，为了吸引用户，可能还需要投资一大笔市场推广预算。 但在一开始令人欣喜的用户高峰期过后，就会出现用户慢慢停用应用的情况。 *这就是 Azure Mobile Engagement 大派用场的时候*：留住用户，让开发人员致力于通过测试和了解慢慢改进应用！

本文所用方法是为了更好地留住用户，提高应用使用率，具体说来就是通过推送通知和应用内消息的方式留住应用用户，但这种方式很特别，其特别之处在于需根据用户在应用中的行为提供适合该用户的消息和通信内容。 本文的目标是确保在适当的时间和适当的地点与适当的受众进行通信。

但是，为了实现这个目标，需首先 *了解用户*，然后根据其行为或特征创建组（本文称为类型），最后再针对每个类型创建相关通信。

## <a name="mobile-engagement-serves-your-objectives"></a>Mobile Engagement 充分满足用户的目标
*目前已讨论了如何留住用户并提高应用使用率，但这样做的目的是什么？*

构建 Mobile Engagement 策略时，需先了解应用的目标和关键绩效指标 (KPI)。

一开始需定义目标和 KPI，这样才能更方便地使用正确的 Prism 定义参与用例。

用例是一个简单的列表，其中包含与用户通信时需要开展的活动，既有简单的欢迎活动，也有由 IT 系统触发的很高级的实用程序通知。 构建良好的用例必须至少包含内容、人员和时间这三部分： 

1. 一个非常短的名称（例如，“欢迎活动”）。
2. **内容**：消息示例（例如，“欢迎加入！ 请注意，登录后第一个月免费试用！”）。 此消息不是一成不变的，可以随时进行更改，但通常需要开门见山地指出所要表述的内容。
3. **人员**：会收到此消息的类型（例如，“所有在 3 天前首次启动应用的用户，虽然访问过登录页面，但并未登录”）。
   * 嗯，通过 Azure Mobile Engagement 可以非常轻松地实现这一点 :)
   * 与前面一样，这个也不是一成不变的，可以随时定义类型，但最好是在确定分类标准时早定义，确保收集到正确的数据。
4. **时间**：活动的时间安排。 可以将活动安排在一个确定的日期，或者安排在根据触发器触发的特定操作之后。 Mobile Engagement 提供很多种选择，可以适当地安排通信时间。

定义用例和类型以后，即可根据相应的准则定义必须在应用程序内收集的数据。 这就需要用到“标记计划”。 标记计划是为了确保向开发人员指定数据收集标准。 有了标记计划，开发人员就能够在 Mobile Engagement 中嵌入正确的设置，使用户能够通过正确的数据开展活动。 此外还需运行测试，确保正确进行了集成并可收集所需数据，这一点很重要。

完成集成并发布应用程序以后，营销人员即可实时查看分析数据，对受众分类，然后开始发送智能型定向推送通知，确保留住应用内外的最终用户。

### <a name="use-cases-to-get-started"></a>用例入门
1. 欢迎策略：根据最终用户在启动应用程序后的行为创建多个推送通知活动，然后在第一次会话后按 D+2/5/10/15 模式再来一轮用户参与活动，提高第一轮的保留率。
2. 根据最终用户的行为推广新内容（功能、文章/视频，或者产品），仅向留住可能性较高的最终用户发送信息。
3. 对应用程序评级：将目标范围限定为用户群中不到 1% 的用户，即在应用商店中对应用的评级最有可能是 5 星的用户。
4. 提高订阅量：将重要内容推广给尚未见过该内容的最终用户，提高订阅量。
5. 教程：不再提供针对所有人的必需教程。 可以编写优秀的应用内教程，仅在特定情况（用户看起来尚未使用过该应用，或在使用某项功能时遇到困难）下通过应用内消息触发这些教程，为什么不这样呢？

## <a name="why-do-you-need-analytics-to-engage"></a>为何需要通过分析来保留用户？
大家现在可能已认识到，仅发送大规模的推送通知是不够的。 确保营销人员和开发人员在适当的时间和适当的地点留住适当的最终用户，这是 Mobile Engagement 的核心概念。 若要了解这三大概念，必须从应用程序收集分析数据，然后通过这些分析数据对受众分类。 还可以对行为分类辅之以其他数据库或 CRM 中的数据，或者通过跨渠道方式获取的数据，更充分地利用此功能。 可以通过 Mobile Engagement 从任何位置检索数据，利用这些数据确定适当的受众。

必须了解最终用户的行为及其实时状态，这样才能最有针对性地留住受众。 营销人员可以通过收集数据重点了解与用例相关的重要事项，实现移动用户参与策略目标。 另外，在实际操作中，最好不要收集分析数据中的所有内容，而只收集那些有意义的内容，对用例有效的内容，这样有助于实现早期设置的目标。 至于如何使用该解决方案、如何确定智能推送通知的目标，从而增强应用程序的用户保留率并确保最终的成功，不妨通过这种方式开展包括启动、尝试、测试和了解在内的一系列工作。

> [!NOTE]
> 请记住：太多数据等于没有数据！

### <a name="use-cases-and-best-practices"></a>用例和最佳实践
以下部分简要介绍一些重要的用例，这些用例是在与客户打交道时遇到过的，适合入门用户。

#### <a name="media"></a>媒体
收集最终用户所使用的那类内容，然后根据相应的行为对受众分类，只向受众发送针对性强且使用可能性较高的那类内容。 这样可避免骚扰整个用户群，确保提高用户保留率。

#### <a name="m-commerce"></a>移动商务
收集应用程序内访问次数最多的产品类别，针对该类别的特定受众发送折扣或新产品促销信息，提高最终用户购买该类产品的可能性。 以提高收入为目标。 再提醒一次，不要发送垃圾邮件！

#### <a name="gaming"></a>游戏
收集最终用户的游戏级别情况，了解其在特定时段消耗在游戏上的时间，确定那些一直停留在当前级别，因此在出现特惠的情况下更有可能跳到下一级别的受众。

提供带奖励的具体活动的消息，鼓励那些有一段时间未玩的用户回到游戏中来。

#### <a name="retail"></a>零售
按受众的喜好或行为收集其更有可能使用的产品或品牌，吸引受众到应用商店购买商品，提高销售收入。

#### <a name="banking"></a>银行
收集首次启动应用程序即创建了帐户的最终用户的数据。 目的是通过定向推送通知部署欢迎策略，提高帐户订阅数。

### <a name="how-to-create-a-great-tag-plan"></a>如何创建优秀的标记计划？
标记计划必须对用户路径进行描述，或者必须是一种应用程序工作流，提供必须收集的所有必需标记（数据），这样才能有足够的分析数据，才能了解用户行为，才能对用户群进行适当的分类。 这不是一个技术性过程。 这样才能让营销人员指定需要根据其 Mobile Engagement 策略收集的数据。

最低要求是至少标记应用程序的所有屏幕（在 Mobile Engagement 中称为“活动”）。 这有助于确定用户路径。

活动可以通过嵌入 *事件* 收集操作信息（例如单击某个按钮）。 这样即可收集应用程序内的交互。 因此，营销人员能够知道用户所访问的屏幕以及所执行的操作。

`Jobs` 属于有一定持续时间的操作。 这对营销人员很有用，例如，可以了解用户创建帐户或登录花了多少时间。 这对开发人员可能也有用，可以监视调用 Web 服务所花的时间。

`Errors` ，了解用户是否存在应用使用问题。 例如，是否频繁出现连接问题。

所有此类数据均可通过参数（在 Mobile Engagement 中称为*额外信息* ）进行强化，这样即可从应用程序收集动态数据。 这对于启用精细分类很重要。 例如，营销人员可以根据用户所使用内容的类型对用户进行分类。 内容的类型属于活动或事件的动态信息。

 是指用于实时确认应用程序状态或用户状态的数据。 此项也可用于对受众群分类，快速进行定向。 例如，可以使用 true/false 状态确定用户是否登录，也可以使用订阅到期日期。

#### <a name="example-of-tags"></a>标记示例
*用例：对受众行为分类，向适当的最终用户发送适当的推送通知内容*

1. 发送推送通知，对某类产品进行促销：根据受众在给定时段内访问过 x 次的产品的类别或其添加到购物车中的特定商品收集行为数据，对受众分类。 可以通过收集的数据对受众分类，然后向适当的受众发送推送通知。
2. 对应用评级：根据受众在社交网络上共享的内容收集数据。 目的是通过确定应用的 *大使* 对受众分类。 使用应用内推送通知时，大使是指应用的最佳受众，可以要求其在应用商店内将应用评为 5 星级。
   
   ![][1]

*用例：声明性数据*

1. 对提醒性新闻进行分类：收集声明性数据，根据受众的偏好对受众分类。 这样即可将特定主题的推送通知发送给真正对其感兴趣的特定受众。
2. 根据登录状态对受众分类。 通过收集数据了解用户是否处于连接状态，或者是否创建了帐户。 这有助于确定尚未登录的最终用户，可发送推送通知鼓励这些最终用户加入。

# <a name="approach-and-process-for-adopting-azure-data-catalog"></a>方法和采用 Azure 数据目录的过程
本文章帮助你开始采用**Azure 数据目录**组织中。 若要成功采用**Azure 数据目录**，你专注于三个关键项目： 展望、 确定组织中的关键业务用例并选择一个试验项目。

## <a name="introducing-the-azure-data-catalog"></a>引入 Azure 数据目录
环境内的工作，已更改的它们应如何能够找到有关数据资产的专家信息的人的预期。 现在，使用大范围的工作区的社交媒体工具，如 Yammer，他们希望能够快速获取各种主题上的帮助和建议。 **Azure 数据目录**帮助企业和团队提供合并到中央存储库中的企业数据资产的相关信息。 数据使用者可以发现这些数据资产，并获得由行业专家提供的知识。

本文提供了一种方法，使用入门**Azure 数据目录**。 本文介绍名为 Adventure Works 的虚构公司的典型的数据目录采用计划。

**什么是 Azure 数据目录？**

**Azure 数据目录**是完全托管的服务在 Azure 和启用自助服务的数据源发现的企业范围的信息 （元数据） 目录中。 与数据目录，你将注册、 发现、 批注，并连接到数据资产。 数据目录旨在管理不同的信息资产，以使它们更轻松地查找数据资产，了解它们，连接到它们。 它可以减少深入了解从可用数据的时间并提高到组织的值。 若要了解详细信息，请参阅[Microsoft Azure 数据目录](https://azure.microsoft.com/services/data-catalog/)。

## <a name="azure-data-catalog-adoption-plan"></a>Azure 数据目录采用计划
**Azure 数据目录**采用计划描述使用服务的好处与利益干系人和用户的通信方式以及什么类型的培训，您提供给用户。 一个关键的成功驱动程序采用数据目录是如何有效地传达给用户和利益干系人的服务的值。 在初始采用计划主要受众是服务的用户。 无论多少购买中你获得利益干系人，如果用户或客户的产品不将其合并到其使用情况，您数据目录采用都不会成功。 因此，本文假定你有利益干系人购买，以及创建的数据目录的用户采用计划的重点。
有效采用计划成功参与人员新增功能，可以使用数据目录，并将这些的信息和指南来实现此目的。 用户需要了解数据目录提供以帮助他们在其作业时都成功的值。 当用户看到数据目录可以帮助他们实现数据的更多结果时，请采用数据目录的值就变得清楚。 更改是更改的硬，，因此有效计划需要考虑的挑战。

采用计划可帮助你通信关键人士成功并实现其目标。 典型的计划解释数据目录将如何使用户生活更容易，并包括以下部分：

* **意向语句**-它可以帮助你确切地与用户和利益干系人讨论采用计划。 它是简要介绍。
* **试验团队和影响因素**-从试点学习团队和影响因素可帮助你优化如何向团队和用户数据目录。 影响因素可以对等指导资深用户。 它还有助于你确定阻止程序问题和采用的驱动程序。
* **规划通信和热门话题**-它可以帮助用户了解如何在数据目录可以帮助他们，和可以培养团队，并最终整个组织内的有机采用。
* **培训计划**-综合性的培训通常会产生采用成功和较好的结果。

## <a name="define-your-data-catalog-project-vision"></a>展望数据目录项目
定义的第一步**Azure 数据目录**采用计划是编写您要完成的启发性说明。 最好是要保留相当全面，愿景陈述尚未足够简洁，若要定义特定短期和长期目标。

下面是一些提示可帮助你定义你愿景：

* **标识部署的关键驱动程序**-从数据目录可以解决的业务需求考虑特定数据源管理。 它可以帮助你状态使用数据目录的顶部优点。 例如，可能有常见的数据源的所有新员工需要了解和使用或只有少数密钥的人深入了解的重要和复杂的数据源。 **Azure 数据目录**可以帮助易于这些数据源发现和理解，以便这些已知的关键点可以直接和早期在中得以解决服务的采用率。
* **为清晰且清晰明了**-清除有关值的同一页面上的愿景的了解获取的每个人数据目录和带来了组织如何愿景支持组织目标。
* **激励人员想要使用数据目录**-你的愿景和沟通计划应激发人员让他们识别数据目录可以受益地查找和连接到数据源，以带来更多价值的数据。
* **指定目标和时间线**-它可以确保你采用计划具有特定的可实现可交付结果。 时间线将保持每个人已设定焦点，并且允许在检查点来衡量成功的。

## <a name="identify-data-catalog-business-use-cases"></a>标识数据目录业务用例
若要标识与数据目录相关的用例，进行与的专家进行各种业务单位来标识相关用例和要解决的业务问题。 查看现有带来挑战人员标识和理解数据资产。 例如，不要团队了解数据资产仅在询问在具有相关的数据源组织中的多个用户后？

最好是选择表示低取得成效的目标的用例： 很重要的情况下尚未高的可能性会成功如果解决了与数据目录。

下面是一些提示来标识用例：

* **定义团队的目标**的团队如何实现其目标？ 由于你想要在此阶段目标尚未不专注于数据目录。 请记住它是有关业务结果，不有关技术。
* **定义业务问题**-有关查找和了解数据资产团队所面临的问题是什么？ 例如，在一个网络文件夹中的 Excel 工作簿中可能找到有关重要的数据源的信息和团队可能支出定位工作簿多长时间。
* **了解与相关更改的团队区域性**-许多采用难题与抵制更改而不是新工具的实现。 团队的响应方式更改很重要，在标识用例，因为现有的进程可能是到位，因为"this is 如何，我们已始终完成"或"如果它东西没出，为什么修复此问题？"。 采用任何新工具或进程时始终最简单的受影响的人员了解要实现从更改，而且非常感谢要解决的问题的重要性的值。
* **保持与数据资产的焦点**-时讨论团队面临的业务问题，需要"穿过杂草"，并专注于来利用多个有效性的企业数据资产的相关度的内容。

## <a name="choose-a-data-catalog-pilot-project"></a>选择一个数据目录试验项目
一个关键成功因素是以简化，并可从简单开始。 具有受约束的作用域定义完善试点有助于保护而陷入困境与项目太复杂，或具有过多的参与者不向前移动的项目。 但它也是重要的用户，从有人怀疑到较早采用混合。 用户接受解决方案帮助你优化未来通信和巴计划。 有人怀疑帮助你识别和解决阻塞问题。 当有人怀疑变得冠军，你可以使用他们的反馈来确定成功驱动程序。

试点计划应阶段在您想要获得与数据目录的业务目标。 当你了解从初始试验，你可以展开你的用户群。 初始的已关闭的试验，最好建立可衡量成功，但最终目标不适用于环保或病毒的增长。 数据目录的有机增长，用户可以控制他们自己的数据使用，并可能会影响和鼓励他人采用和贡献目录。

### <a name="target-the-right-team"></a>目标合适的团队
当你选择试验项目时，选择最吸引人的方案，解决了现有的业务问题的团队。 例如，业务分析人员可从 SQL Server 数据库创建报告。 问题是，她之后感知的数据源仅的多个同事通话。 最后之后会浪费时间尝试查找要使用的数据源，, 她找到有关 Excel 工作簿，其中包含每个数据源的说明。 虽然 Excel 工作簿充分介绍他/她需要的表，但是她将具有快速找到这些数据源如果他们已注册并将批注中**Azure 数据目录**。

### <a name="identify-data-heroes"></a>标识数据英雄
第一个试点项目应具有少数几个人生成数据和使用数据，以便团队有平衡表示形式。

**数据创建器**是通过有关数据源的专业技能的人员。 例如，在另一个团队 David 了合作，广泛与密钥 Adventure Works 数据源。 之前的采用率**Azure 数据目录**，David 已创建 Excel 工作簿来捕获有关 Adventure Works 数据源信息。

**数据使用者**是通过使用这些数据上的专业技能的人员来解决业务问题。 例如，Nancy 是业务分析师使用 Adventure Works SQL Server 数据源来分析数据。

业务问题之一， **Azure 数据目录**解决了是将连接**数据创建器**到**数据使用者**。 它通过充当企业数据源信息的中央存储库来执行此操作。 使用数据目录，David 注册 Adventure Works 和 SQL Server 数据源。 使用 crowdsourcing 发现此数据源的任何用户可以共享上的数据，除了使用她发现的数据的她意见。 例如，Nancy 通过搜索目录中，发现数据源，并共享其有关的数据方面的专业的知识。  现在，其他组织中获得共享知识库搜索数据目录。

* 若要了解有关注册数据源的详细信息，请参阅[注册数据源](data-catalog-get-started.md)。
* 若要了解有关发现数据源的详细信息，请参阅[搜索数据源](data-catalog-get-started.md)。

### <a name="start-small-and-focused"></a>启动小型和已设定焦点
对于大多数企业试验项目，则应设置为种子与高价值的数据源目录，以便业务用户可以快速查看数据目录的值。 IT 是标识常见的数据源将你的试点团队感兴趣的良好开端。 支持的数据源，例如 SQL Server 中，我们建议使用**Azure 数据目录**数据源注册工具。 利用数据源注册工具，你可以注册大量的数据源包括 SQL Server 和 Oracle 数据库和 SQL Server Reporting Services 报表。 当前数据源的完整列表，请参阅[Azure 数据目录支持的数据源](data-catalog-dsr.md)。

一旦你标识并注册密钥的数据源，则可以也导入在其他位置存储的数据源说明。 数据目录 API 允许开发人员从另一个位置，例如 Excel 工作簿 David 创建和维护加载说明和批注。

### <a name="an-example-project"></a>示例项目
对于此示例，Nancy 业务分析人员创建她的团队，使用 SQL Server 数据库中的数据的报表。 问题是，她之后感知的数据源仅的多个同事通话。 她将具有快速找到这些数据源如果他们已注册并将在一个中心位置如批注**Azure 数据目录**。

为了说明 Nancy 和她团队可以如何轻松地找到高价值的数据，你使用数据源注册工具填充有关的信息 （元数据） 的数据源目录。 这种方式数据库有关的信息可供团队和企业，而不只需几个人。 一旦在数据目录中注册数据源，Nancy 和她团队可以轻松地使用它们。 结果是她的团队和企业版中的更全面且相关数据目录。 多个团队采用数据目录，如业务数据源变得更容易地查找和使用;因此，启用要实现与你的数据的更多以数据为中心的区域性。

若要了解有关数据源注册工具的详细信息，请参阅[开始使用 Azure 数据目录](data-catalog-get-started.md)。

作为试验项目的一部分，Nancy 的团队也使用该 David 的 Excel 工作簿中进行了描述的数据源，同事维护。 企业中的其他团队还使用 Excel 工作簿来描述数据源，由于 IT 团队决定创建一个工具，可将 Excel 工作簿迁移到数据目录。 通过使用数据目录 REST API 导入现有的批注，试验项目团队可以从使用数据源注册工具，完成，但以前通过数据生产者和使用者，而无需手动重新输入记录的信息的数据源中提取的元数据包含完整的数据目录。 随着企业数据目录的增长，组织可以使用数据源注册工具，常见的数据源和数据目录 API 的自定义源和常见方案。

### <a name="execute"></a>Execute
此时您已用例的数据目录，并在你确定你的第一个项目。 此外，你已注册的密钥的 Adventure Works 数据源并已添加信息从现有 Excel 工作簿使用工具它生成。 现在，它是时间和试点团队合作，开始数据目录采用过程的工作。

下面是一些提示可帮助你入门：

* **创建刺激**-业务用户获取兴奋他们认为，如果**Azure 数据目录**生活更加轻松。 尝试将围绕该解决方案，它提供的好处不技术会话键。
* **促进更改**-从小和规划传达给业务用户。 若要成功，至关重要涉及从开始处的用户，以便它们影响结果并开发一种解决方案有关的所有权。
* **梳理早期采用者**-早期采用者是热衷它们执行的操作，并且非常高兴地宣传的好处的业务用户**Azure 数据目录**发给其对等方。
* **面向培训**-业务用户不需要了解有关数据目录的所有内容，因此面向培训，以解决特定的团队目标。 专注于用户做什么，以及如何他们的某些任务可能会更改，以合并**Azure 数据目录**到其每日的例程。
* **愿意失败**-如果试点未达到所需的结果，重新计算，并确定区域以更改-然后移到较大作用域在试点中修复问题。

试点团队跳转到使用的数据目录之前，计划会议以讨论预期为试验项目，并提供初始的培训。

### <a name="set-expectations"></a>设置预期
设置异常和目标有助于业务用户专注于特定的可交付结果。 若要使项目保持正轨，分配常规 (例如： 每天或每周基于的作用域和试点的持续时间) 家庭作业。 数据目录的最有价值的功能之一是 crowdsourcing 数据资产，以便业务用户可以得益于的企业数据的知识。 对于每个试点团队成员，若要注册或批注用过的至少一个数据源是一个很好的作业分配。 请参阅[注册数据源](data-catalog-get-started.md)和[如何批注数据源](data-catalog-get-started.md)。

与按定期计划以查看一些批注团队会面。 有关数据源的良好批注是成功的数据目录采用的核心，因为它们提供有意义的数据源 insights 在一个中心位置。 如果没有良好批注，有关数据源的知识将保持在整个企业分散。 请参阅[如何批注数据源](data-catalog-get-started.md)。

而且，ultimate 测试的项目是用户是否能够发现和了解他们需要使用的数据源。 试点用户应定期进行测试以确保它们工作中使用的一天到一天的数据源相关的目录。 缺少或不正确批注所需的数据源时，它应作为一条提醒，若要注册更多数据源，或者提供附加的批注。 这种做法不仅将值添加到试点的工作量，但还生成试点完成后，通过对其他团队执行的有效习惯。

### <a name="provide-training"></a>提供的培训
培训应足以让用户启动，并且定制到特定目标和体验的试点团队成员的级别。 若要开始使用培训，可以按照中的步骤[开始使用 Azure 数据目录](data-catalog-get-started.md)文章。 此外，你可以下载[Azure 数据目录试点项目定型演示文稿](https://github.com/Azure-Samples/data-catalog-dotnet-get-started/blob/master/Azure%20Data%20Catalog%20Training.pptx?raw=true)。 此 PowerPoint 演示文稿应帮助你开始向你的试点团队成员引入数据目录。

## <a name="conclusion"></a>结语
一旦相当平稳运行试点团队并且已实现你的初始目标，您应扩展到多个团队的数据目录采用。 应用，并优化你从试验项目以在你的组织展开数据目录的了解。

试点参加早期采用者可有助于获取有关采用数据目录的好处。 它们可以共享与其他小组数据目录是如何帮助其团队解决业务问题、 更轻松地发现数据源和共享有关他们使用的数据源的见解。 例如，在 Adventure Works 试点团队的早期采用者可以显示其他若要查找有关曾难以找到和了解的 Adventure Works 数据资产的信息是多么容易。

此文章是有关如何开始使用**Azure 数据目录**组织中。 我们希望你以前可以启动数据目录的试验项目，然后展开在你的组织的数据目录。

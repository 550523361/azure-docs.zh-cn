# <a name="overview-of-tls-termination-and-end-to-end-tls-with-application-gateway"></a>使用应用程序网关 TLS 端接和端到端 TLS 概述

传输层安全 （TLS）以前称为安全套接字层 （SSL），是建立 Web 服务器和浏览器之间加密链接的标准安全技术。 此链接可确保在 Web 服务器与浏览器之间传递的所有数据都会得到保密和加密。 应用程序网关支持网关的 TLS 端接以及端到端 TLS 加密。

## <a name="tls-termination"></a>TLS 终止

应用程序网关支持在网关处终止 TLS，之后流量通常未加密地流向后端服务器。 在应用程序网关执行 TLS 端接有许多优点：

- **改进的性能**– 执行 TLS 解密时的最大性能冲击是初始握手。 为了提高性能，执行解密的服务器缓存 TLS 会话 I 并管理 TLS 会话票证。 如果此操作是在应用程序网关上执行的，则来自同一个客户端的所有请求都可以使用缓存的值。 如果在后端服务器上完成，则每次客户端的请求转到其他服务器时，客户端必须重新进行身份验证。 使用 TLS 票证有助于缓解此问题，但并非所有客户端都支持 TLS 票证，并且配置和管理这些票证可能有难度。
- **更好地利用后端服务器**– SSL/TLS 处理非常需要 CPU，并且随着密钥大小的增加而变得更加密集。 从后端服务器中消除此任务可使它们专注于以最有效的方式提供内容。
- **智能路由**– 通过解密流量，应用程序网关可以访问请求内容（如标头、URI 等），并可以使用此数据路由请求。
- **证书管理**– 证书只需在应用程序网关上购买和安装，而不需要在所有后端服务器上安装。 这可以节省时间和开支。

要配置 TLS 端接，需要向侦听器添加 TLS/SSL 证书，以使应用程序网关能够按照 TLS/SSL 协议规范派生对称密钥。 然后，可以使用该对称密钥来加密和解密发送到网关的流量。 TLS/SSL 证书需要以个人信息交换 （PFX） 格式使用。 此文件格式适用于导出私钥，后者是应用程序网关对流量进行加解密所必需的。

> [!NOTE] 
>
> 应用程序网关不会提供任何用于创建新证书，或者将证书请求发送到证书颁发机构的功能。

对于 TLS 连接正常工作，您需要确保 TLS/SSL 证书满足以下条件：

- 当前日期和时间处于证书上的“Valid from”（有效起始日期）和“Valid to”（有效结束日期）日期范围内。
- 证书的“公用名”(CN) 与请求中的主机头匹配。 例如，如果客户端向 `https://www.contoso.com/` 发出请求，则 CN 必须为 `www.contoso.com`。

### <a name="certificates-supported-for-tls-termination"></a>TLS 终止支持的证书

应用程序网关支持以下类型的证书：

- CA（证书颁发机构）证书：CA 证书是由证书颁发机构 （CA） 颁发的数字证书
- EV（扩展验证）证书：EV 证书是符合行业标准证书准则的证书。 这将将浏览器定位器栏变为绿色，并发布公司名称。
- 通配符证书：此证书支持基于 *.site.com的任意数量的子域，您的子域将在此替换 *。 但是，它不支持 site.com，因此，如果用户在不键入前导“www”的情况下访问你的网站，则通配符证书不会反映这一点。
- 自签名证书：客户端浏览器不信任这些证书，并将警告用户虚拟服务的证书不是信任链的一部分。 自签名证书适合用于测试，或者管理员会在其中控制客户端并且可以安全绕过浏览器安全警报的环境。 切勿将自签名证书用于生产工作负荷。

有关详细信息，请参阅[使用应用程序网关配置 TLS 端接](https://docs.microsoft.com/azure/application-gateway/create-ssl-portal)。

### <a name="size-of-the-certificate"></a>证书大小
检查[应用程序网关限制](https://docs.microsoft.com/azure/azure-resource-manager/management/azure-subscription-service-limits#application-gateway-limits)部分，了解支持的最大 TLS/SSL 证书大小。

## <a name="end-to-end-tls-encryption"></a>端到端 TLS 加密

某些客户可能不希望与后端服务器进行未加密的通信。 这可能是因为安全要求、符合性要求，或者应用程序可能仅接受安全连接。 对于此类应用程序，应用程序网关支持端到端 TLS 加密。

端到端 TLS 允许您安全地将敏感数据传输到加密的后端，同时仍利用应用程序网关提供的第 7 层负载平衡功能的优势。 部分功能包括：基于 Cookie 的会话相关性、基于 URL 的路由、基于站点的路由支持，或注入 X-Forwarded-* 标头。

当配置端到端 TLS 通信模式时，应用程序网关将终止网关上的 TLS 会话并解密用户流量。 然后，它会应用配置的规则，以选择要将流量路由到的适当后端池实例。 然后，应用程序网关启动到后端服务器的新 TLS 连接，并使用后端服务器的公钥证书重新加密数据，然后再将请求传输到后端。 来自 Web 服务器的任何响应都会经历相同的过程返回最终用户。 通过在[后端 HTTP 设置](https://docs.microsoft.com/azure/application-gateway/configuration-overview#http-settings)中设置协议设置到 HTTPS，从而启用端到端 TLS，然后应用于后端池。

TLS 策略适用于前端和后端流量。 在前端上，应用程序网关充当服务器并强制实施该策略。 在后端，应用程序网关充当客户端，并在 TLS 握手期间将协议/密码信息作为首选项发送。

应用程序网关只会与下面所述的后端实例通信：已将其证书加入应用程序网关的允许列表，或者其证书已由已知的 CA 颁发机构签名，并且证书 CN 与 HTTP 后端设置中的主机名匹配。 这些实例包括受信任的 Azure 服务，例如 Azure 应用服务 Web 应用和 Azure API 管理。

如果后端池中成员的证书未由众所周知的 CA 颁发机构签名，则必须使用证书配置后端池中启用端到端 TLS 的每个实例以允许安全通信。 添加证书后，可确保应用程序网关仅与已知后端实例通信。 从而进一步保护端到端通信。

> [!NOTE] 
>
> Azure 应用服务 Web 应用和 Azure API 管理等受信任的 Azure 服务不需要身份验证证书设置。

> [!NOTE] 
>
> 添加到**后端 HTTP 设置**以对后端服务器进行身份验证的证书可以与在应用程序网关为 TLS 终止而添加到**侦听器**的证书相同，也可以为增强安全性而不同。

## <a name="end-to-end-tls-and-whitelisting-of-certificates"></a>端到端 TLS 和证书白名单

应用程序网关只会与已知的后端实例通信，这些实例已将其证书加入应用程序网关的允许列表。 要启用证书允许列表，必须将后端服务器证书的公钥上传到应用程序网关（而不是根证书）。 然后仅允许连接到已知且加入允许列表的后端。 剩余的后端会导致网关错误。 自签名证书仅用于测试目的，不建议用于生产工作负荷。 如前面的步骤中所述，此类证书必须加入应用程序网关的允许列表，才可以使用。

> [!NOTE]
> Azure 应用服务等受信任的 Azure 服务不需要身份验证证书设置。

## <a name="end-to-end-tls-with-the-v2-sku"></a>使用 v2 SKU 端到端 TLS

身份验证证书已弃用，并由应用程序网关 v2 SKU 中的受信任的根证书替换。 它们的功能与身份验证证书类似，但有一些主要区别：

- 由众所周知的 CA 颁发机构签名的证书，其 CN 与 HTTP 后端设置中的主机名匹配，不需要任何额外的步骤才能端到端 TLS 工作。 

   例如，如果后端证书由知名 CA 颁发并具有 contoso.com 的 CN，并且后端 http 设置的主机字段也设置为 contoso.com，则不需要其他步骤。 您可以将后端 http 设置协议设置为 HTTPS，并且运行状况探测和数据路径都将启用 TLS。 如果使用 Azure 应用服务或其他 Azure Web 服务作为后端，则这些服务也隐式受信任，并且不需要执行端到端 TLS 的进一步步骤。
   
> [!NOTE] 
>
> 为了使 TLS/SSL 证书受信任， 后端服务器的证书必须由包含在应用程序网关的受信任存储中的 CA 颁发。 如果证书不是由受信任的 CA 颁发的，则应用程序网关将检查颁发 CA 的证书是否由受信任的 CA 颁发，等等，直到找到受信任的 CA（此时将建立受信任的安全连接）或找不到受信任的 CA（此时应用程序网关将标记后端）。 因此，建议后端服务器证书同时包含根和中间的 C。

- 如果证书是自签名的，或者由未知中介签名，那么要在 v2 SKU 中启用端到端 TLS，必须定义受信任的根证书。 应用程序网关仅与符合以下条件的后端进行通信：后端的服务器证书的根证书与池关联的后端 http 设置中的受信任根证书列表之一匹配。

> [!NOTE] 
>
> 自签名证书必须是证书链的一部分。 V2 SKU 不支持无链的单个自签名证书。

- 除了根证书匹配之外，应用程序网关还验证后端 http 设置中指定的主机设置是否与后端服务器 TLS/SSL 证书提供的常见名称 （CN） 的主机设置匹配。 尝试建立后端的 TLS 连接时，应用程序网关将服务器名称指示 （SNI） 扩展设置为后端 http 设置中指定的主机。
- 如果**从后端地址选择主机名**而不是后端 http 设置中的主机字段，则 SNI 标头始终设置为后端池 FQDN，后端服务器 TLS/SSL 证书上的 CN 必须与其 FQDN 匹配。 此方案不支持具有 IP 的后端池成员。
- 根证书是来自后端服务器证书的 base64 编码的根证书。
